== Introduction

Modern cloud-native applications often need to handle high-volume, real-time
data streams and respond to events as they occur, especially in domains such
as IoT, financial services, and e-commerce. Traditional request-response
architectures struggle with the demands of event-driven systems that require
asynchronous, non-blocking communication between services.

{reactive-messaging-spec-name} simplifies the development of asynchronous,
message-driven applications by providing a framework for building reactive
systems that handle backpressure, process continuous data streams, and
integrate seamlessly with message brokers such as Apache Kafka and RabbitMQ
(via AMQP). By leveraging reactive streams and CDI beans, developers can
create scalable, resilient microservices that embrace event-driven
architecture patterns.

This chapter covers {reactive-messaging-spec-name}
{reactive-messaging-spec-version} as part of MicroProfile {mp-version}.
All examples require JDK 21 or later and a MicroProfile {mp-version}
compatible runtime such as Open Liberty, Quarkus, or Payara.

{reactive-messaging-spec-name} {reactive-messaging-spec-version} requires the
`jakarta.*` namespace introduced in Jakarta EE 9.1 and is not compatible with
applications using the legacy `javax.*` namespace from Java EE or earlier
Jakarta EE releases.

== Event-Driven Architecture (EDA) Overview

Event-Driven Architecture (EDA) is a software design paradigm where system
behaviour is driven by events. Events are significant changes in state or
occurrences that components need to respond to. Unlike traditional
request-response architectures, components in an event-driven system
communicate asynchronously by producing and consuming events, decoupling
producers from consumers and allowing each to scale and evolve independently.

=== Principles and Benefits

Event-Driven Architecture is built on several core principles that
directly translate into system-level benefits.

Asynchronous communication::
  Components communicate through events without waiting for immediate
  responses, enabling non-blocking operations and allowing systems to
  handle more concurrent requests with improved responsiveness.

Loose coupling::
  Producers emit events without knowing who will consume them, and
  consumers react without knowing the event origin. This independence
  allows new consumers to be added without modifying existing producers,
  making the system more adaptable to changing requirements.

Scalability::
  EDA naturally supports horizontal scaling. Multiple consumers can
  process events in parallel, each at its own pace, optimising resource
  utilisation without introducing contention between components.

Resilience::
  Because components are loosely coupled and communicate asynchronously,
  failures in one component do not directly cascade to others. Message
  brokers can persist events, allowing consumers to process them once
  they recover.

Real-time processing::
  Events can be processed as they occur, enabling real-time analytics,
  monitoring, and responsive user experiences.

=== Comparison with Request-Response Architecture

Traditional request-response architecture follows a synchronous, call-and-wait
pattern where a client sends a request and blocks until it receives a response.
This model has significant limitations in distributed systems:

[cols="2,3,3", options="header"]
|===
|Aspect |Request-Response |Event-Driven

|Communication style
|Synchronous, blocking
|Asynchronous, non-blocking

|Coupling
|Tight coupling between client and server
|Loose coupling between producers and consumers

|Scalability
|Limited by synchronous dependencies
|Highly scalable with parallel processing

|Failure handling
|Failures directly impact the caller
|Failures are isolated; events can be reprocessed

|Data flow
|Point-to-point
|Publish-subscribe or streaming

|Latency
|Lower latency for simple operations
|Optimised for high throughput and batch processing
|===

.Order processing: request-response vs event-driven
====
In a request-response model, when a user places an order, the order service
synchronously calls the payment service, inventory service, and shipping
service in sequence. If any service is slow or fails, the entire operation
is delayed or fails.

In an event-driven model, placing an order emits an `OrderCreated` event.
The payment, inventory, and shipping services independently consume this
event and process it asynchronously. If one service is slow, it does not
block the others, and each can retry independently.
====

=== Common Use Cases

Event-Driven Architecture excels in scenarios requiring high performance,
scalability, and real-time processing.

High-frequency event processing::
  Stock trading platforms process millions of trades per second using
  non-blocking, asynchronous event pipelines. Gaming backends apply the
  same pattern to handle real-time player actions and state updates across
  distributed game servers.

Real-time data processing::
  IoT deployments collect and analyse data from thousands of sensors as
  events arrive. Live analytics dashboards aggregate and visualise metrics
  from streaming data, and fraud detection systems analyse transaction
  events in real time to identify suspicious patterns before they escalate.

Microservices communication::
  Order processing systems coordinate multiple services — inventory,
  payment, and shipping — through a shared event stream rather than
  synchronous service calls. Notification systems use the same pattern to
  broadcast events to multiple subscribers such as email, SMS, and push
  notification services.

Data integration::
  Change data capture streams database changes to analytics platforms or
  search indexes as they occur. ETL pipelines apply transformations to
  data streams between systems without batch windows or scheduled jobs.

For Maven projects, add the MicroProfile Reactive Messaging API dependency to your `pom.xml`:

[source,xml]
----
<dependency>
    <groupId>org.eclipse.microprofile.reactive.messaging</groupId>
    <artifactId>microprofile-reactive-messaging-api</artifactId>
    <version>3.0.1</version>
    <scope>provided</scope>

----

The `provided` scope indicates that the runtime environment (such as Open Liberty, Quarkus, or Payara) will supply the implementation. If you're building a standalone application, change the scope to `compile`.

[source,groovy]

dependencies {
    providedCompile 'org.eclipse.microprofile.reactive.messaging:microprofile-reactive-messaging-api:3.0.1'
}
----

== Introduction to Reactive Streams

Reactive Streams is a specification that provides a standard for asynchronous stream processing with non-blocking backpressure. It defines a minimal set of interfaces, methods, and protocols to enable different libraries and tools to interoperate while handling streams of data efficiently.

The Reactive Streams specification addresses a critical challenge in stream processing: how to handle scenarios where data producers generate data faster than consumers can process it. Without proper flow control, this mismatch can lead to out-of-memory errors, dropped messages, or system instability.

=== Core Interfaces: Publisher and Subscriber

The Reactive Streams specification defines four core interfaces that work together to enable asynchronous stream processing with backpressure control:

*Publisher*:

A Publisher is a provider of a potentially unbounded number of sequenced elements, publishing them according to the demand received from its Subscribers.

[source,java]
----
public interface Publisher<T> {
    void subscribe(Subscriber<? super T> subscriber);
}
----

The Publisher produces elements and pushes them to subscribers. It respects backpressure by only emitting elements when subscribers signal demand through the Subscription.

*Subscriber*:

A Subscriber receives and processes elements from a Publisher. It implements callbacks for different lifecycle events:

[source,java]
----
public interface Subscriber<T> {
    void onSubscribe(Subscription subscription);
    void onNext(T item);
    void onError(Throwable throwable);
    void onComplete();
}
----

* `onSubscribe`: Called when the subscription is established, providing a Subscription object that the Subscriber uses to control the flow of elements.
* `onNext`: Called for each element produced by the Publisher. The Subscriber processes the element synchronously before this method returns.
* `onError`: Called if an error occurs during stream processing, terminating the stream. No further elements will be received.
* `onComplete`: Called when the Publisher has finished sending all elements successfully, signaling normal stream termination.

*Subscription Lifecycle*:

The interaction between Publisher and Subscriber follows a defined protocol:

1. A Subscriber subscribes to a Publisher by calling `publisher.subscribe(subscriber)`
2. The Publisher calls `subscriber.onSubscribe(subscription)`, passing a Subscription object
3. The Subscriber uses `subscription.request(n)` to request elements
4. The Publisher sends up to `n` elements via `subscriber.onNext(item)`
5. When the stream ends, the Publisher calls either `subscriber.onComplete()` (success) or `subscriber.onError(throwable)` (failure)

=== Subscription and Processor

*Subscription*:

A Subscription represents the link between a Publisher and a Subscriber. It provides methods for requesting elements and canceling the subscription:

[source,java]
----
public interface Subscription {
    void request(long n);
    void cancel();
}
----

* `request(n)`: The Subscriber requests up to `n` elements from the Publisher. This is the core mechanism for backpressure control—subscribers pull data at their own pace rather than having data pushed to them.
* `cancel()`: The Subscriber can cancel the subscription to stop receiving elements. This is useful when the subscriber has received enough data or needs to shut down gracefully.

The request-based pull model enables backpressure: subscribers control the flow by requesting only as many elements as they can handle, preventing buffer overflow and resource exhaustion.

*Processor*:

A Processor sits between a Publisher and a Subscriber, acting as both. It receives elements from an upstream Publisher, processes or transforms them, and publishes the results to downstream Subscribers:

[source,java]
----
public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {
}
----

Processors enable stream pipelines where data flows through multiple transformation stages. Each Processor can apply operations like filtering, mapping, aggregating, or enriching data before passing results downstream.

*Example*: A Processor might receive temperature readings in Celsius from a sensor Publisher, convert them to Fahrenheit, and publish the converted values to downstream Subscribers that update dashboards or trigger alerts.

*Backpressure in Action*:

[source,java]
----
// Subscriber requests 10 elements initially
subscription.request(10);

// Publisher sends up to 10 elements via onNext
for (int i = 0; i < 10; i++) {
    subscriber.onNext(data[i]);
}

// After processing, Subscriber requests more when ready
subscription.request(5);
----

This request-based flow control prevents overwhelming slow consumers. If a Subscriber is processing data slowly (perhaps writing to a database), it requests small batches. Fast consumers can request larger batches for higher throughput. The Publisher never sends more elements than requested, ensuring efficient resource utilization across the entire pipeline.

MicroProfile Reactive Messaging builds on the Reactive Streams specification, providing a higher-level, annotation-driven API that leverages these concepts while hiding much of the complexity through CDI integration. Developers work with simple annotated methods rather than implementing these interfaces directly.

== Defining Messaging Logic with CDI Beans

MicroProfile Reactive Messaging leverages Contexts and Dependency Injection (CDI) to provide a declarative, annotation-based approach to building reactive messaging applications. By using CDI beans, developers can define message processing logic in a clean, maintainable way without dealing with low-level messaging APIs.

=== Basic Structure of Messaging Beans

A messaging bean in MicroProfile Reactive Messaging is a standard CDI bean with methods annotated to indicate how they participate in message processing. These beans are automatically discovered and initialized by the CDI container.

*Key Characteristics*:

* *CDI Managed*: Beans are managed by the CDI container, enabling dependency injection and lifecycle management.
* *Annotation-Driven*: Methods are annotated with `@Incoming`, `@Outgoing`, or both to define message flows.
* *Flexible Scopes*: Beans can use various CDI scopes (@ApplicationScoped, @RequestScoped, etc.) based on requirements.

*Basic Bean Structure*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Incoming;
import org.eclipse.microprofile.reactive.messaging.Outgoing;
import jakarta.enterprise.context.ApplicationScoped;

@ApplicationScoped
public class PriceConverter {
    
    @Incoming("prices-in")
    @Outgoing("prices-out")
    public double convertPrice(int priceInCents) {
        return priceInCents / 100.0;
    }
}
----

In this example:
* The bean is application-scoped, meaning one instance exists for the entire application.
* The `convertPrice` method receives messages from the "prices-in" channel.
* It processes each message (converting cents to dollars).
* It sends the result to the "prices-out" channel.

=== Leveraging CDI for Dependency Injection and Scope Management

CDI provides powerful capabilities that enhance messaging beans:

*Dependency Injection*:

Messaging beans can inject other services, repositories, or resources needed for processing:

[source,java]
----
@ApplicationScoped
public class OrderProcessor {
    
    @Inject
    OrderService orderService;
    
    @Inject
    NotificationService notificationService;
    
    @Incoming("orders")
    public CompletionStage<Void> processOrder(Order order) {
        return orderService.createOrder(order)
            .thenAccept(savedOrder -> 
                notificationService.sendConfirmation(savedOrder)
            );
    }
}
----

*Scope Management*:

// Updated for MicroProfile Reactive Messaging 3.0.1 specification
Different CDI scopes affect bean lifecycle and threading. The specification mandates support for the following CDI scopes:

* *@ApplicationScoped*: Single instance for the entire application. Suitable for stateless processors. Most common for messaging beans.
* *@Dependent*: New instance for each injection point. Use with caution as it can create multiple message processing pipelines.

Implementations can support other scopes, such as `@RequestScoped`, but behavior outside the two mandated scopes is not defined by the specification.

*Configuration Injection*:

MicroProfile Config can be injected to externalize configuration:

[source,java]
----
@ApplicationScoped
public class ConfigurableProcessor {
    
    @Inject
    @ConfigProperty(name = "processor.batch.size", defaultValue = "100")
    int batchSize;
    
    @Incoming("items")
    @Outgoing("batches")
    public PublisherBuilder<List<Item>> batchItems(PublisherBuilder<Item> items) {
        // Batch items using configured batchSize
        return items.collect(
            () -> new ArrayList<Item>(),
            (list, item) -> {
                list.add(item);
                return list.size() < batchSize;
            }
        );
    }
}
----

*Lifecycle Integration*:

CDI lifecycle callbacks can be used with messaging beans:

[source,java]
----
@ApplicationScoped
public class MessageHandler {
    
    private ExecutorService executor;
    
    @PostConstruct
    public void initialize() {
        executor = Executors.newFixedThreadPool(10);
    }
    
    @Incoming("tasks")
    public CompletionStage<Void> handleTask(Task task) {
        return CompletableFuture.runAsync(() -> 
            processTask(task), executor
        );
    }
    
    @PreDestroy
    public void cleanup() {
        executor.shutdown();
    }
}
----

*Benefits of CDI Integration*:

* *Simplified Code*: No need to manually manage bean lifecycles or wire dependencies.
* *Testability*: CDI beans can be easily mocked and tested in isolation.
* *Modularity*: Messaging logic can be organized across multiple beans with clear separation of concerns.
* *Integration*: Seamlessly works with other Jakarta EE and MicroProfile specifications.

== Understanding Data Streaming and Event Sourcing

Modern applications often need to handle continuous flows of data and maintain
comprehensive audit trails of state changes. Data streaming and event sourcing
are two complementary patterns that address these needs in event-driven
architectures.

=== Data Streaming

Data streaming involves processing a continuous flow of data as it arrives,
rather than in discrete batches. Unlike traditional batch processing, where
data is collected and processed at scheduled intervals, streaming systems
process each event or message as it occurs, enabling near-real-time insights
and actions.

Continuous processing::
  Data is processed immediately as it arrives. Streams have no defined end;
  they represent ongoing, unbounded sequences of events rather than finite
  data sets.

Low latency::
  Processing occurs with minimal delay, typically in milliseconds, making
  streaming suitable for time-sensitive workloads where batch windows are
  not acceptable.

Scalable throughput::
  Streaming systems distribute processing across multiple nodes to handle
  high-volume data flows, allowing throughput to scale horizontally as
  demand increases.

Common use cases for data streaming include near-real-time analytics such as
aggregating website clickstream data or sensor telemetry, event processing for
fraud detection and system monitoring, and data pipeline workloads such as
synchronizing databases or populating search indexes.

.E-commerce clickstream example
====
An e-commerce platform streams user clickstream data to analyse shopping
behaviour as it occurs, enabling personalised product recommendations and
dynamic pricing without waiting for nightly batch reports.
====

{reactive-messaging-spec-name} supports data streaming through its
integration with messaging systems such as Apache Kafka, providing a
declarative programming model for building streaming applications on top
of MicroProfile {mp-version} runtimes.

=== Using Event Sourcing for Storing Changes

Event sourcing is a pattern where all changes to application state are stored as a sequence of events, rather than storing only the current state. Instead of updating a record in place, the system appends events describing what happened.

*Core Concepts of Event Sourcing*:

* *Event Store*: A database or log that stores all events in the order they occurred. Common implementations include dedicated event stores (EventStoreDB, Apache Kafka) or relational databases with an events table.
* *Immutable Events*: Once written, events cannot be changed, providing an authoritative audit trail. Events are facts about what happened, not intentions or commands.
* *Event Replay*: The current state can be reconstructed by replaying events from the beginning or from the last snapshot.
* *Aggregate*: A domain object that processes commands and produces events. The aggregate enforces business rules before emitting events.

*Benefits of Event Sourcing*:

* *Complete Audit Trail*: Every state change is recorded with full context (timestamp, user, reason), enabling compliance with regulations like PCI DSS, SOX, and GDPR.
* *Temporal Queries*: You can query what the state was at any point in time by replaying events up to that moment.
* *Event Replay*: Bugs in business logic can be fixed and events replayed to correct state without data loss.
* *Multiple Projections*: Different views or read models can be created from the same event stream, supporting CQRS (Command Query Responsibility Segregation) patterns.
* *Natural Integration*: Events can be published to message brokers for reactive processing by other microservices.

*Challenges and Considerations*:

* *Complexity*: Event sourcing adds architectural complexity compared to traditional CRUD operations.
* *Event Schema Evolution*: As the system evolves, you need strategies to handle old event versions (upcasting, versioning).
* *Performance*: Replaying thousands of events to rebuild state can be slow. Snapshots (periodic state checkpoints) mitigate this.
* *Eventual Consistency*: Read models derived from events may lag behind writes, requiring careful UX design.

*Example*: In a banking application, instead of storing just the current account balance in a single row, the event-sourced system stores immutable events:

----
AccountOpened(accountId: "123", initialBalance: 1000, timestamp: "2024-01-01T10:00:00Z")
AccountCredited(accountId: "123", amount: 100, timestamp: "2024-01-02T14:30:00Z")
AccountDebited(accountId: "123", amount: 50, timestamp: "2024-01-03T09:15:00Z")
----

The current balance (1050) is computed by replaying all events. If you need the balance as of January 2nd, replay events up to that date (1100). The event store becomes the authoritative source of truth, while the current balance is a projection that can be rebuilt at any time.

*Integration with MicroProfile*: Event sourcing pairs naturally with MicroProfile capabilities for building reactive, auditable microservices:

* *MicroProfile Reactive Messaging*: When domain events are persisted to the event store, they can be simultaneously published to Kafka topics using `@Outgoing` channels. Other microservices subscribe to these events with `@Incoming` to maintain their own projections or trigger downstream workflows. For example, when an `AccountDebited` event is stored, it's published to a Kafka topic where a fraud detection service consumes it to check for suspicious patterns.

* *MicroProfile GraphQL*: Event streams can be exposed through GraphQL subscriptions, allowing clients to receive real-time updates as events occur. Temporal queries ("what was the account state on January 2nd?") can be implemented as GraphQL queries that replay events up to a specified timestamp.

* *MicroProfile Fault Tolerance*: Apply `@Retry` and `@CircuitBreaker` patterns when persisting events to ensure the event store remains consistent even during transient failures. Since events are immutable, idempotency strategies ensure duplicate event writes are safely handled.

* *MicroProfile Telemetry*: Instrument event processing with tracing spans to track event flow from command receipt through event persistence and projection updates. Emit metrics on event throughput, replay performance, and projection lag to monitor system health.

This combination enables microservices to maintain complete audit trails while staying responsive and resilient through asynchronous event processing.

=== Rebuilding State from Events

One of the powerful features of event sourcing is the ability to rebuild application state by replaying events. This process involves:

1. *Starting with an Initial State*: Begin with an empty or default state for the aggregate or projection.
2. *Applying Events Sequentially*: Process each event in order, updating the state according to the event's semantics.
3. *Reaching the Current State*: After processing all events, the system arrives at the current state.

*Benefits of State Rebuilding*:

* *Disaster Recovery*: If the current state is lost or corrupted, it can be reconstructed from events without data loss.
* *New Projections*: New read models or views can be created by replaying events with different logic, enabling new features without migrating data.
* *Debugging and Testing*: Historical states can be recreated to understand bugs or test scenarios. Developers can replay production events in development environments to reproduce issues.
* *Event Schema Migration*: When event formats change, replay events through migration logic to update projections to the new schema.

*Optimization with Snapshots*:

For systems with long event histories, replaying every event can be time-consuming. Snapshots periodically capture the current state at a specific event sequence number, allowing rebuilding to start from the snapshot and replay only subsequent events. This dramatically reduces recovery time for aggregates with thousands of events.

The snapshot strategy balances storage costs against replay performance. Common approaches include snapshotting every N events (e.g., every 100 events) or after significant state changes (e.g., order completion).

*Example*: An order management system stores events in sequence:

----
OrderCreated(orderId: "789", customerId: "C123", timestamp: "2024-01-15T09:00:00Z")
ItemAdded(orderId: "789", productId: "P456", quantity: 2, timestamp: "2024-01-15T09:05:00Z")
ItemAdded(orderId: "789", productId: "P789", quantity: 1, timestamp: "2024-01-15T09:07:00Z")
PaymentProcessed(orderId: "789", amount: 150.00, timestamp: "2024-01-15T10:30:00Z")
OrderShipped(orderId: "789", trackingNumber: "TRK123", timestamp: "2024-01-16T14:00:00Z")
----

To rebuild the current state of order "789", replay these events in sequence, updating the order's status and contents at each step. The final state shows a shipped order with two line items and confirmed payment.

*Integration with MicroProfile*:

* *MicroProfile Reactive Messaging for Event Replay*: Consume events from Kafka topics using `@Incoming` channels to build projections. When creating a new projection, use Kafka consumer group offsets to replay events from the beginning of the topic. For example, a reporting service can subscribe to order events and build a sales dashboard by processing the entire event history, then continue receiving live updates.

* *MicroProfile Config for Snapshot Strategy*: Configure snapshot intervals using MicroProfile Config properties such as `snapshot.interval.events=100` or `snapshot.enabled=true`. This allows operations teams to tune snapshot frequency based on system performance without code changes.

* *MicroProfile Metrics for Replay Monitoring*: Track replay progress with metrics like `events_replayed_total`, `replay_duration_seconds`, and `projection_lag_events` to monitor how far projections lag behind the event stream. These metrics help identify when snapshots are needed or when projection logic needs optimization.

* *MicroProfile Fault Tolerance during Replay*: Apply `@Retry` and `@Timeout` to event replay operations to handle transient failures when reading from event stores. Use `@Bulkhead` to limit concurrent replay operations and prevent resource exhaustion when rebuilding multiple projections simultaneously.

This combination of MicroProfile specifications enables robust, observable event sourcing implementations. Event stores like Apache Kafka provide durable, ordered event streams, while MicroProfile Reactive Messaging handles consumption and processing. MicroProfile Config, Metrics, and Fault Tolerance add production-ready operational capabilities around event replay and projection management.

== @Channel and @Emitter Support

While declarative messaging with `@Incoming` and `@Outgoing` handles most use cases, there are scenarios where more programmatic control over message production is needed. MicroProfile Reactive Messaging provides `@Channel` and `@Emitter` for these situations.

=== Named Channels for Message Flow

Channels are named pathways through which messages flow in a Reactive Messaging application. They serve as logical conduits connecting message producers, processors, and consumers.

*Channel Types*:

* *Internal Channels*: Connect components within the same application. Messages flow in-memory without external broker involvement.
* *External Channels*: Connected to external message brokers (Kafka, AMQP, JMS, etc.) via connectors configured through MicroProfile Config.

*Channel Configuration*:

Channels are configured using MicroProfile Config, typically in `microprofile-config.properties` or as environment variables:

[source,properties]
----
# Incoming channel from Kafka
mp.messaging.incoming.orders.connector=smallrye-kafka
mp.messaging.incoming.orders.topic=order-topic
mp.messaging.incoming.orders.bootstrap.servers=localhost:9092
mp.messaging.incoming.orders.group.id=order-service

# Outgoing channel to Kafka
mp.messaging.outgoing.notifications.connector=smallrye-kafka
mp.messaging.outgoing.notifications.topic=notification-topic
mp.messaging.outgoing.notifications.bootstrap.servers=localhost:9092
----

*Channel Names*:

Channel names referenced in `@Incoming`, `@Outgoing`, and `@Channel` annotations must match the configuration property names:

[source,java]
----
@Incoming("orders")  // Matches mp.messaging.incoming.orders.*
public void processOrder(Order order) {
    // Processing logic
}
----

Internal channels do not require connector configuration. They are created automatically when a method produces to a channel that another method consumes from within the same application.

=== Using @Channel to Access Channels

The `@Channel` annotation allows developers to inject a channel directly into a bean, providing programmatic access to the message stream. This is useful for observing message flows or integrating reactive streams with imperative code:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Channel;
import io.smallrye.mutiny.Multi;
import jakarta.inject.Inject;

@ApplicationScoped
public class OrderService {
    
    @Inject
    @Channel("orders")
    Multi<Order> orders;
    
    public void displayOrders() {
        orders.subscribe().with(
            order -> System.out.println("Order: " + order.getId())
        );
    }
}
----

The injected type depends on the stream's cardinality:

* `Multi<T>` for streams of multiple elements
* `Publisher<T>` for Reactive Streams Publisher
* `PublisherBuilder<T>` for Reactive Streams operators

=== Using @Emitter for Programmatic Message Emission

An `Emitter` provides a way to programmatically send messages to a channel from imperative code, such as REST endpoints, scheduled tasks, or event handlers. This bridges imperative and reactive programming models:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Emitter;
import org.eclipse.microprofile.reactive.messaging.Channel;
import jakarta.inject.Inject;

@ApplicationScoped
public class OrderController {
    
    @Inject
    @Channel("orders")
    Emitter<Order> orderEmitter;
    
    public void createOrder(Order order) {
        orderEmitter.send(order);
    }
}
----

*Emitter Features*:

* *Payload Sending*: `send(T payload)` sends a payload and returns `CompletionStage<Void>`. The `CompletionStage` completes when the emitted message is acknowledged. If processing fails, the `CompletionStage` completes exceptionally.
* *Message Sending*: `send(Message<T> message)` sends a `Message` object and returns `void`. Use this form when you need custom acknowledgment or negative-acknowledgment functions.
* *Backpressure Aware*: Emitters respect backpressure from downstream consumers, preventing overwhelming slow processors.

*Emitter with Payload Acknowledgment*:

[source,java]
----
@Inject
@Channel("orders")
Emitter<Order> orderEmitter;

public CompletionStage<Void> createOrderAsync(Order order) {
    return orderEmitter.send(order)
        .thenAccept(v -> log.info("Order sent successfully"))
        .exceptionally(throwable -> {
            log.error("Failed to send order", throwable);
            return null;
        });
}
----

*Emitter with Message and Custom Acknowledgment*:

[source,java]
----
@Inject
@Channel("orders")
Emitter<Order> orderEmitter;

public void createOrderWithAck(Order order) {
    orderEmitter.send(Message.of(order,
        () -> {
            log.info("Order acknowledged: " + order.getId());
            return CompletableFuture.completedFuture(null);
        },
        throwable -> {
            log.error("Order nacked: " + order.getId(), throwable);
            return CompletableFuture.completedFuture(null);
        }
    ));
}
----

*Use Cases for @Emitter*:

* *REST Endpoints*: Converting synchronous REST requests into messages for asynchronous processing.
* *Scheduled Tasks*: Periodically generating messages based on `@Scheduled` timers or cron expressions.
* *External Integrations*: Forwarding events from external systems (webhooks, file watchers) to message channels.
* *Event Sourcing*: Publishing domain events to event stores or Kafka topics after persisting them.
* *Testing*: Injecting test messages into channels programmatically during integration tests.

*Example: REST to Messaging Bridge*:

[source,java]
----
import jakarta.ws.rs.*;
import jakarta.ws.rs.core.MediaType;
import jakarta.ws.rs.core.Response;

@Path("/orders")
@ApplicationScoped
public class OrderResource {

    @Channel("orders")

    }
}
----

This pattern decouples the REST layer from the processing layer, enabling asynchronous processing and better scalability. The REST endpoint returns immediately while the order processing happens asynchronously in a separate messaging pipeline.

=== Controlling Backpressure with @OnOverflow

The `@OnOverflow` annotation controls what happens when messages are emitted faster than downstream consumers can process them. Without this annotation, the default strategy is `OnOverflow.Strategy.BUFFER` with a buffer size of 128 elements (or the value of the `mp.messaging.emitter.default-buffer-size` configuration property).

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Channel;
import org.eclipse.microprofile.reactive.messaging.Emitter;
import org.eclipse.microprofile.reactive.messaging.OnOverflow;
import jakarta.inject.Inject;

@Inject
@Channel("orders")
@OnOverflow(value = OnOverflow.Strategy.BUFFER, bufferSize = 300)
Emitter<Order> orderEmitter;
----

*Available Overflow Strategies*:

[cols="1,3"]
|===
|Strategy |Behavior

|`BUFFER`
|Buffer messages up to `bufferSize` elements. If `bufferSize` is not set, the `mp.messaging.emitter.default-buffer-size` property is used (default: 128). Throws `java.lang.IllegalStateException` from `send()` when the buffer is full.

|`UNBOUNDED_BUFFER`
|Use an unbounded buffer. Messages accumulate in memory without limit. The application may run out of memory if messages accumulate faster than they are consumed. Use only when you can guarantee consumption keeps pace.

|`THROW_EXCEPTION`
|Throw `IllegalStateException` from `send()` immediately if the downstream cannot keep up. No buffering occurs.

|`DROP`
|Drop the most recently emitted message when the downstream cannot keep up. Previously buffered messages are retained. This strategy prioritizes older messages.

|`FAIL`
|Propagate a failure signal downstream when backpressure is detected. The emitter stops accepting new messages and the stream terminates.

|`LATEST`
|Keep only the most recent message, discarding all previously buffered messages when the downstream cannot keep up. This strategy prioritizes the newest data.

|`NONE`
|Ignore backpressure signals completely. The downstream consumer must implement its own buffering strategy. Use with caution as this can lead to unbounded memory growth.
|===

*Choosing the Right Strategy*:

* *`BUFFER`*: Best for most scenarios where occasional slowdowns are acceptable and you want to preserve all messages.
* *`DROP` or `LATEST`*: Appropriate for high-frequency sensor data or metrics where missing some values is acceptable.
* *`FAIL`*: Use when any backpressure indicates a system problem that should halt processing.
* *`THROW_EXCEPTION`*: Provides immediate feedback to the caller that the system is overloaded.

[source,java]
----
// For high-frequency sensor data where latest value matters most
@Inject
@Channel("temperature-readings")
@OnOverflow(OnOverflow.Strategy.LATEST)
Emitter<Temperature> temperatureEmitter;

// For critical transactions that must all be processed
@Inject
@Channel("payments")
@OnOverflow(value = OnOverflow.Strategy.BUFFER, bufferSize = 1000)
Emitter<Payment> paymentEmitter;
----

When the `@OnOverflow` annotation is absent, the `BUFFER` strategy with default buffer size applies automatically. This provides a sensible default for most applications while allowing fine-grained control when needed.

== Dealing with Backpressure

Backpressure is one of the most critical challenges in reactive systems. It occurs when message producers generate data faster than consumers can process it. Without proper handling, this mismatch can lead to memory exhaustion, dropped messages, or system failures.

=== Understanding Backpressure in Messaging Context

In a messaging system, backpressure scenarios include:

* *Fast Producer, Slow Consumer*: A Kafka topic receives thousands of messages per second, but the consumer can only process 100 per second.
* *Bursty Traffic*: Periodic spikes in message volume overwhelm downstream processors.
* *Resource Constraints*: Database connections, network bandwidth, or CPU become bottlenecks.

*Traditional Approaches and Problems*:

1. *Unbounded Buffers*: Queue all messages in memory, leading to OutOfMemoryError.
2. *Dropping Messages*: Discard excess messages, causing data loss.
3. *Blocking*: Block producers until consumers catch up, reducing throughput and causing cascading failures.

Reactive Streams' request-based flow control provides a better solution: consumers explicitly request the number of elements they can handle, and producers respect these requests.

=== Automatic Handling of Backpressure

MicroProfile Reactive Messaging automatically manages backpressure through the Reactive Streams protocol:

*Request Propagation*:

When using this pattern:

[source,java]
----
@Incoming("data")
@Outgoing("processed-data")
public String process(String data) {
    return transform(data);
}
----

The framework:
1. Requests one element from the upstream channel
2. Waits for the `process` method to complete
3. Sends the result downstream
4. Requests the next element

This sequential processing naturally provides backpressure: the upstream producer will not send more data until the current item is processed.

*Asynchronous Processing with Backpressure*:

[source,java]
----
@Incoming("tasks")
@Outgoing("results")
public CompletionStage<Result> processAsync(Task task) {
    return CompletableFuture.supplyAsync(() -> 
        expensiveOperation(task)
    );
}
----

The framework waits for the CompletionStage to complete before requesting the next message, ensuring backpressure is maintained even with async operations.

=== Strategies for Configuring Backpressure Behavior

While automatic backpressure works well for many cases, developers can configure behavior for specific requirements:

*Buffering Strategy*:

Configure buffer sizes for channels:

[source,properties]
----
mp.messaging.incoming.data.buffer-size=256
----

This creates a buffer of 256 messages, allowing the producer to send ahead while the consumer processes at its own pace.

*Overflow Strategy*:

Define what happens when buffers fill:

[source,properties]
----
mp.messaging.incoming.data.overflow-strategy=drop-oldest
----

Strategies include:
* `drop-oldest`: Remove the oldest message when buffer is full
* `drop-latest`: Discard the newest message
* `fail`: Throw an exception

*Concurrent Processing*:

Process multiple messages concurrently while maintaining backpressure:

[source,java]
----
@Incoming("tasks")
@Outgoing("results")
public PublisherBuilder<Result> processConcurrent(
        PublisherBuilder<Task> tasks) {
    return tasks.flatMapCompletionStage(3, task ->
        CompletableFuture.supplyAsync(() -> process(task))
    );
}
----

The `flatMapCompletionStage(3, ...)` processes up to 3 messages concurrently, providing controlled parallelism while respecting backpressure.

*Batching*:

Process messages in batches to improve throughput:

[source,java]
----
@Incoming("events")
@Outgoing("batched-events")
public PublisherBuilder<List<Event>> batch(
        PublisherBuilder<Event> events) {
    return events.collect(
        () -> new ArrayList<Event>(),
        (list, event) -> {
            list.add(event);
            return list.size() < 100;
        }
    );
}
----

*Throttling*:

Limit message processing rate:

[source,java]
----
@Incoming("high-volume")
@Outgoing("throttled")
public PublisherBuilder<Data> throttle(PublisherBuilder<Data> data) {
    return ReactiveStreams.fromPublisher(
        Multi.createFrom().publisher(data.buildRs())
            .group().intoLists().of(10, Duration.ofSeconds(1))
            .flatMap(list -> Multi.createFrom().iterable(list))
    );
}
----

== Message Acknowledgments and Error Handling

Reliable message processing requires proper acknowledgment and error handling strategies. MicroProfile Reactive Messaging provides flexible mechanisms to ensure messages are processed reliably and failures are handled appropriately.

=== Message Acknowledgment Strategies

Acknowledgment confirms that a message has been successfully processed. The timing and handling of acknowledgments affect reliability and performance.

*Acknowledgment Strategies*:

MicroProfile Reactive Messaging supports several acknowledgment strategies configured via `@Acknowledgment`:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Acknowledgment;
----

*1. PRE_PROCESSING*:

Acknowledge the message before processing:

[source,java]
----
@Incoming("data")
@Acknowledgment(Acknowledgment.Strategy.PRE_PROCESSING)
public void process(String data) {
    // Message already acknowledged
    processData(data);
}
----

* *Use when*: Processing is idempotent or occasional message loss is acceptable.
* *Risk*: If processing fails, the message is already acknowledged and will not be redelivered.

*2. POST_PROCESSING*:

Acknowledge after successful processing (default for most signatures):

[source,java]
----
@Incoming("data")
@Acknowledgment(Acknowledgment.Strategy.POST_PROCESSING)
public void process(String data) {
    processData(data);
    // Message acknowledged after method returns
}
----

* *Use when*: You need at-least-once delivery guarantees.
* *Behavior*: If processing throws an exception, the message is not acknowledged.

*3. MANUAL*:

Manually control when to acknowledge:

[source,java]
----
@Incoming("data")
@Acknowledgment(Acknowledgment.Strategy.MANUAL)
public CompletionStage<Void> process(Message<String> message) {
    return processData(message.getPayload())
        .thenCompose(v -> message.ack())
        .exceptionally(throwable -> {
            message.nack(throwable);
            return null;
        });
}
----

* *Use when*: You need fine-grained control over acknowledgment timing.
* *Methods*: `message.ack()` acknowledges, `message.nack(throwable)` negative acknowledges.

*4. NONE*:

No automatic acknowledgment:

[source,java]
----
@Incoming("data")
@Acknowledgment(Acknowledgment.Strategy.NONE)
public void process(String data) {
    // No acknowledgment occurs
}
----

* *Use when*: Connected to systems that do not support acknowledgment.

=== Error Handling and Failure Strategies

When message processing fails, strategies are needed to handle the failure:

*1. Retry*:

Automatically retry failed messages:

[source,properties]
----
mp.messaging.incoming.data.failure-strategy=retry
mp.messaging.incoming.data.retry.max-retries=3
mp.messaging.incoming.data.retry.delay=1000
----

*2. Dead Letter Queue*:

Send failed messages to a separate channel:

[source,properties]
----
mp.messaging.incoming.data.failure-strategy=dead-letter-queue
mp.messaging.incoming.data.dead-letter-queue.topic=failed-data
----

*3. Ignore*:

Log and ignore failures:

[source,properties]
----
mp.messaging.incoming.data.failure-strategy=ignore
----

*Programmatic Error Handling*:

[source,java]
----
@Incoming("orders")
public CompletionStage<Void> processOrder(Message<Order> message) {
    return orderService.process(message.getPayload())
        .thenCompose(v -> message.ack())
        .exceptionally(throwable -> {
            if (throwable instanceof ValidationException) {
                log.warn("Invalid order", throwable);
                return message.ack(); // Acknowledge to skip
            } else {
                log.error("Processing failed", throwable);
                return message.nack(throwable); // Retry
            }
        });
}
----

*Circuit Breaker Integration*:

Combine with MicroProfile Fault Tolerance:

[source,java]
----
@Incoming("external-calls")
@CircuitBreaker(requestVolumeThreshold = 10)
@Fallback(fallbackMethod = "fallbackProcess")
public CompletionStage<Void> process(Data data) {
    return externalService.call(data);
}

public CompletionStage<Void> fallbackProcess(Data data) {
    log.warn("Circuit open, using fallback");
    return CompletableFuture.completedFuture(null);
}
----

=== Using @Blocking for Worker Threads

// Updated for MicroProfile Reactive Messaging 3.0.1 specification
NOTE: The `@Blocking` annotation is not defined in the MicroProfile Reactive Messaging specification. It is an extension provided by the SmallRye Reactive Messaging implementation (used by Quarkus). Other runtime implementations may provide equivalent mechanisms. Consult your runtime's documentation for portability guidance.

By default, message processing occurs on the event loop thread. For blocking operations (database calls, file I/O, synchronous HTTP calls), use `@Blocking` to offload to a worker thread:

[source,java]
----
import io.smallrye.reactive.messaging.annotations.Blocking;

@ApplicationScoped
public class DatabaseProcessor {
    
    @Inject
    EntityManager entityManager;
    
    @Incoming("user-updates")
    @Blocking
    public void updateUser(User user) {
        // Blocking database operation
        entityManager.merge(user);
        entityManager.flush();
    }
}
----

*@Blocking Parameters*:

[source,java]
----
@Blocking(value = "my-pool", ordered = false)
----

* `value`: Custom worker pool name
* `ordered`: Whether to preserve message order (default: true)

*When to Use @Blocking*:
* Database operations (JPA, JDBC)
* File system operations
* Synchronous HTTP clients
* CPU-intensive computations

*When NOT to Use @Blocking*:
* Asynchronous operations returning CompletionStage
* Non-blocking I/O operations
* Simple, fast transformations

            }
        };
    }

== Best Practices and Design Patterns

// Updated for MicroProfile Reactive Messaging 3.0.1 specification
=== Assembly and Deployment Validation

When the application starts, the Reactive Messaging implementation connects all annotated methods, `Emitter` instances, `@Channel` injections, and connectors, then validates the resulting graph. The implementation throws a `DeploymentException` if any of the following conditions exist:

* A method annotated with `@Incoming` has no upstream channel.
* A method annotated with `@Outgoing` has no downstream channel.
* A method annotated with `@Incoming` has multiple upstream channels.
* A method annotated with `@Outgoing` has multiple downstream channels.
* An `Emitter` has no downstream channel.
* An `Emitter` has multiple downstream channels.
* An injected `@Channel` has no upstream channel.
* An injected `@Channel` has multiple upstream channels.
* The application references a connector that is not available.
* An incoming connector has no downstream channel or multiple downstream channels.
* An outgoing connector has no upstream channel or multiple upstream channels.

NOTE: Each channel name may be used in at most one `@Incoming` annotation across the entire application. If two separate methods carry `@Incoming` with the same channel name, the implementation treats the channel as having multiple downstream consumers, which is not supported and causes a `DeploymentException` at startup. The same constraint applies to `@Outgoing`: a given channel name may appear in at most one `@Outgoing` annotation.

Building production-ready reactive messaging applications requires following best practices and established design patterns.

=== Handling Idempotent Operations

Messages may be delivered more than once. Ensure operations are idempotent:

*Idempotency Key Pattern*:

[source,java]
----
@Incoming("payments")
@Blocking
public void processPayment(Message<Payment> message) {
    Payment payment = message.getPayload();
    String idempotencyKey = payment.getIdempotencyKey();
    
    if (processedPayments.contains(idempotencyKey)) {
        log.info("Payment already processed: {}", idempotencyKey);
        message.ack();
        return;
    }
    
    try {
        paymentService.process(payment);
        processedPayments.add(idempotencyKey);
        message.ack();
    } catch (Exception e) {
        message.nack(e);
    }
}
----

*Database-Based Idempotency*:

[source,java]
----
@Transactional
public void processOrder(Order order) {
    Optional<ProcessedOrder> existing = 
        repository.findByExternalId(order.getExternalId());
    
    if (existing.isPresent()) {
        return; // Already processed
    }
    
    // Process order
    ProcessedOrder processed = new ProcessedOrder(order);
    repository.persist(processed);
}
----

== What's New in MicroProfile Reactive Messaging 3.0.1

This section summarizes the changes introduced in MicroProfile Reactive Messaging 3.0 and the 3.0.1 patch release, and their impact on existing applications.

=== Jakarta EE 9.1 Alignment

MicroProfile Reactive Messaging 3.0.1 aligns with Jakarta EE 9.1. All `javax.*` package imports must be updated to their `jakarta.*` equivalents. Applications that depend on `javax.enterprise`, `javax.inject`, or other `javax.*` packages are not compatible with this version without migration.

*Migration action*: Replace all `javax.*` imports with `jakarta.*` counterparts throughout the application.

=== API Fix: Message.of(T, ack, nack) Default Acknowledgement

MicroProfile Reactive Messaging 3.0 includes a fix for an issue where acknowledgement did not work correctly when using the three-argument `Message.of(T, ack, nack)` factory method (issue https://github.com/eclipse/microprofile-reactive-messaging/issues/143[#143]). In prior versions, the default acknowledgement logic could be silently ignored in certain processing scenarios.

*Migration action*: Review all usage of `Message.of(T, Supplier<CompletionStage<Void>>, Function<Throwable, CompletionStage<Void>>)`. Verify that acknowledgement and negative-acknowledgement callbacks execute as expected with the updated implementation.

=== Emitter.send() Return Type Clarification

The `Emitter` interface defines two distinct `send` overloads with different return types:

* `send(T payload)` returns `CompletionStage<Void>`, which completes when the message is acknowledged or fails exceptionally when processing fails.
* `send(Message<T> message)` returns `void`. Use this overload when supplying custom acknowledgement functions on the `Message` object.

*Migration action*: Update any code that calls `send(Message<T>)` and incorrectly assigns or chains its return value as a `CompletionStage`.

=== TCK Compatibility with CDI 4

The Technology Compatibility Kit (TCK) tests were updated to be compatible with CDI 4. This change does not affect the public API but is relevant for runtime implementors and for applications running on runtimes that support MicroProfile 7.1 (which requires CDI 4).

=== Breaking Changes

The sole breaking change in this release is the required Jakarta EE 9.1 namespace migration. Applications that use the `javax.*` namespace cannot run on a MicroProfile Reactive Messaging 3.0.1-compliant runtime without code changes.
