== Introduction

Modern cloud-native applications often need to process high-volume, real-time data and react to events as they occur. Traditional request-response communication is not well-suited for long-running, asynchronous interactions between services.

{reactive-messaging-spec-name} provides an annotation-driven programming model for building asynchronous, message-driven applications with support for backpressure. In practice, applications connect to brokers such as Kafka or AMQP-based systems by using runtime-specific connectors.

Using this model, developers can build scalable and resilient microservices that communicate through events rather than tightly coupled synchronous calls.

== Event-Driven Architecture (EDA) Overview

Event-Driven Architecture (EDA) is a software design paradigm in which system behavior is driven by events.

An event represents a meaningful change in state that other components can observe and process. For example, when an order is placed, one service can emit an event that payment, inventory, and shipping services react to independently.

Components communicate asynchronously by producing and consuming events. This loose coupling makes the system more flexible, scalable, and resilient than a purely request-response approach. EDA is particularly well-suited to distributed systems, microservices, and real-time applications.

=== Comparison with Request-Response Architecture

Traditional request-response architecture follows a synchronous, call-and-wait pattern where a client sends a request and blocks until it receives a response.
This model has significant limitations in distributed systems:

[cols="2,3,3", options="header"]
|===
|Aspect |Request-Response |Event-Driven

|Communication style
|Synchronous, blocking
|Asynchronous, non-blocking

|Coupling
|Tight coupling between client and server
|Loose coupling between producers and consumers

|Scalability
|Limited by synchronous calls and blocking threads
|Highly scalable with asynchronous processing and backpressure

|Failure handling
|Failures directly impact the caller
|Failures are more isolated; messages can often be retried or reprocessed depending on the broker and configuration

|Data flow
|Point-to-point request/response
|Publish-subscribe, queues, or streams

|===

== Reactive Streams

Reactive Streams defines a standard for asynchronous stream processing with backpressure. It enables producers and consumers to coordinate data flow safely and efficiently.

Backpressure prevents producers from overwhelming consumers by allowing consumers to request only the amount of data they are ready to process.

MicroProfile Reactive Messaging builds on this model and provides an annotation-driven programming style using concepts such as channels, `@Incoming`, and `@Outgoing`.

For learning, local development, and testing, many runtimes provide an internal in-memory connector. This connector allows messages to flow between channels within the same application, without requiring an external broker. For production systems, use a messaging provider such as Apache Kafka or RabbitMQ.

=== Connector Dependencies

MicroProfile Reactive Messaging requires connectors to integrate with messaging systems such as Apache Kafka, AMQP, or JMS. Connectors are runtime-specific implementations that handle communication with the underlying messaging infrastructure. They allow your application to produce and consume messages without dealing directly with low-level protocol details.

The exact connector dependency depends on the runtime you are using. For example, when using Open Liberty with Kafka, configure the connector in `microprofile-config.properties`:

[source,properties]
----
mp.messaging.connector.liberty-kafka.bootstrap.servers=localhost:9092
mp.messaging.incoming.orders.connector=liberty-kafka
mp.messaging.incoming.orders.topic=order-topic
----

Depending on your runtime and packaging model, you may also need to make the Kafka client libraries available to the server or application.

Refer to your runtime's documentation for the correct connector artifact coordinates, supported brokers, and configuration details.

== Defining Messaging Logic with CDI Beans

MicroProfile Reactive Messaging leverages Contexts and Dependency Injection (CDI) to define message processing logic through declarative annotations on CDI beans. This approach keeps messaging code clean and maintainable without requiring developers to work directly with low-level messaging APIs or manually implement Reactive Streams interfaces.

=== Basic Structure of Messaging Beans

A messaging bean in MicroProfile Reactive Messaging is a standard CDI bean with methods annotated to indicate how they participate in message processing. These beans are discovered by CDI, and the reactive messaging runtime wires the annotated methods into the message flow.

*Key Characteristics*:

* *CDI Managed*: Beans are managed by the CDI container, enabling dependency injection and lifecycle management.
* *Annotation-Driven*: Methods are annotated with `@Incoming`, `@Outgoing`, or both to define message flows.
* *Flexible Scopes*: Beans can use various CDI scopes such as `@ApplicationScoped`, `@RequestScoped`, or `@Dependent`, depending on the use case.

*Basic Bean Structure*:

[source,java]
----
import io.microprofile.tutorial.store.order.entity.Order;
import io.microprofile.tutorial.store.order.entity.OrderStatus;
import jakarta.enterprise.context.ApplicationScoped;
import org.eclipse.microprofile.reactive.messaging.Incoming;
import org.eclipse.microprofile.reactive.messaging.Outgoing;

@ApplicationScoped
public class PaymentRequestHandler {

    @Incoming("orders")
    @Outgoing("paid-orders")
    public Order processPayment(Order order) {
        order.setStatus(OrderStatus.PAID);
        return order;
    }
}
----

In this example:

* The bean is application-scoped, so one instance serves the application.
* The `processPayment` method receives `Order` messages from the `orders` channel.
* It performs a simple transformation by updating the order status to `PAID`.
* It sends the updated order to the `paid-orders` channel for downstream processing.

=== Leveraging CDI for Dependency Injection and Scope Management

CDI provides powerful capabilities that enhance messaging beans:

*Dependency Injection*:

Messaging beans can inject other services, repositories, or resources needed for processing:

[source,java]
----
@ApplicationScoped
public class OrderProcessor {
    
    @Inject
    OrderService orderService;
    
    @Inject
    NotificationService notificationService;
    
    @Incoming("orders")
    public CompletionStage<Void> processOrder(Order order) {
        return orderService.createOrder(order)
            .thenAccept(savedOrder -> 
                notificationService.sendConfirmation(savedOrder)
            );
    }
}
----

*Scope Management*:

Different CDI scopes affect bean lifecycle and threading. The specification mandates support for the following CDI scopes:

* *@ApplicationScoped*: Single instance for the entire application. Suitable for stateless processors. Most common for messaging beans.
* *@Dependent*: New instance for each injection point. Use with caution as it can create multiple message processing pipelines.

Implementations can support other scopes, such as `@RequestScoped`, but behavior outside the two mandated scopes is not defined by the specification.

*Benefits of CDI Integration*:

* *Simplified Code*: No need to manually manage bean lifecycles or wire dependencies.
* *Testability*: CDI beans can be easily mocked and tested in isolation.
* *Modularity*: Messaging logic can be organized across multiple beans with clear separation of concerns.
* *Integration*: Seamlessly works with other Jakarta EE and MicroProfile specifications.

== Channels and Wiring

=== Named Channels for Message Flow

Channels are named pathways through which messages flow in a Reactive Messaging application. They serve as logical conduits connecting message producers, processors, and consumers.

A typical flow in the MicroProfile e-commerce store application now looks like this:

* the `shoppingcart` component submits an order to the existing `order` service
* the `order` service endpoint accepts order creation requests and publishes an `OrderCreated` event to a channel `orders`, that downstream services can consume asynchronously.
* `payment`, `inventory`, and `shipment` components can subscribe to that event and react independently
* additional processors can filter, enrich, or route the same event stream without changing the public REST API

*Channel Types*:

* *Internal Channels*: Connect components within the same application when methods are wired together by channel name.
* *External Channels*: Connected to external message brokers (Kafka, AMQP, JMS) via connectors configured through MicroProfile Config.

*Channel Configuration*:

Channels are configured using MicroProfile Config, typically in `microprofile-config.properties` or as environment variables:

[source,properties]
----
# Incoming channel from Kafka
mp.messaging.incoming.orders.connector=smallrye-kafka
mp.messaging.incoming.orders.topic=order-topic
mp.messaging.incoming.orders.bootstrap.servers=localhost:9092
mp.messaging.incoming.orders.group.id=order-service

# Outgoing channel to Kafka
mp.messaging.outgoing.notifications.connector=smallrye-kafka
mp.messaging.outgoing.notifications.topic=notification-topic
mp.messaging.outgoing.notifications.bootstrap.servers=localhost:9092
----

*Channel Names*:

Channel names referenced in `@Incoming`, `@Outgoing`, and `@Channel` annotations must match the configuration property names:

[source,java]
----
@Incoming("orders")  // Matches mp.messaging.incoming.orders.*
public void processOrder(Order order) {
    // Processing logic
}
----

Internal channels do not require connector configuration. For communication between separate services or multiple server instances, configure an external broker such as Kafka.

== Declarative Messaging with @Incoming and @Outgoing

Declarative messaging is at the heart of MicroProfile Reactive Messaging. By using simple annotations, developers can define complex message processing pipelines without writing boilerplate code for message handling, threading, or backpressure management. The framework handles these concerns automatically based on the method signatures.

=== Consuming Messages with @Incoming

The `@Incoming` annotation marks a method as a consumer of messages from a specified channel. When messages arrive on that channel, the runtime automatically invokes the method.

*Basic Message Consumption*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Incoming;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.inject.Inject;

@ApplicationScoped
public class PaymentRequestHandler {
    
    @Inject
    PaymentService paymentService;
    
    @Incoming("orders")
    public void processOrderCreated(Order order) {
        paymentService.requestPayment(order);
    }
}
----

In the store application, the method receives the `Order` payload published after order creation and hands it off to the payment step asynchronously. The runtime automatically acknowledges messages after the method completes successfully. If an exception is thrown, the message is negatively acknowledged and can be retried based on the connector's retry policy.

The consumer can delegate the actual payment step to the payment service:

[source,java]
----
import io.microprofile.tutorial.store.order.entity.Order;
import io.microprofile.tutorial.store.order.entity.OrderStatus;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.inject.Inject;
import org.eclipse.microprofile.config.inject.ConfigProperty;

import java.util.concurrent.CompletableFuture;
import java.util.concurrent.CompletionStage;

@ApplicationScoped
public class PaymentService {

    @Inject
    @ConfigProperty(name = "payment.gateway.endpoint",
        defaultValue = "https://payment-gateway.example/authorize")
    String endpoint;

    public CompletionStage<Order> requestPayment(Order order) {
        return CompletableFuture.supplyAsync(() -> {
            order.setStatus(OrderStatus.PAID);
            return order;
        });
    }
}
----

Reactive consumers often hand off to existing domain services instead of embedding all business logic in the messaging method itself.

*Supported Method Signatures*:

MicroProfile Reactive Messaging supports various method signatures to accommodate different processing patterns:

*1. Simple Payload (Automatic Acknowledgment)*:

[source,java]
----
@Incoming("items")
public void process(String item) {
    // Process item
    // Message is automatically acknowledged after method returns
}
----

*2. Asynchronous Processing with CompletionStage*:

[source,java]
----
@Incoming("items")
public CompletionStage<Void> processAsync(String item) {
    return processItemAsync(item);
    // Message acknowledged when CompletionStage completes
}
----

*3. Message Wrapper for Manual Control*:

[source,java]
----
@Incoming("items")
public CompletionStage<Void> process(Message<String> message) {
    String payload = message.getPayload();
    
    return processItem(payload)
        .thenCompose(v -> message.ack())  // Manual acknowledgment
        .exceptionally(throwable -> {
            message.nack(throwable);  // Negative acknowledgment on failure
            return null;
        });
}
----

*4. Reactive Streams Processing*:

[source,java]
----
@Incoming("items")
public Multi<String> process(Multi<String> items) {
    return items
        .map(String::toUpperCase)
        .filter(item -> item.length() > 3);
}
----

The choice of signature depends on your processing requirements:

* Use a simple payload for synchronous, blocking operations.
* Use `CompletionStage` for asynchronous operations.
* Use `Message` wrapper when you need manual acknowledgment control or access to metadata.
* Use reactive types (`Multi`, `Publisher`) for stream transformations.

=== Accessing Message Metadata

Messages can carry metadata in addition to their payload. This metadata might include Kafka headers, timestamps, partition keys, or connector-specific information.

*Accessing Metadata through Message Wrapper*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Message;
import io.smallrye.reactive.messaging.kafka.api.IncomingKafkaRecordMetadata;

@Incoming("orders")
public CompletionStage<Void> processOrder(Message<Order> message) {
    Order order = message.getPayload();
    
    // Access Kafka-specific metadata
    message.getMetadata(IncomingKafkaRecordMetadata.class)
        .ifPresent(metadata -> {
            String topic = metadata.getTopic();
            int partition = metadata.getPartition();
            long offset = metadata.getOffset();
            long timestamp = metadata.getTimestamp();
            
            log.info("Processing order from topic={}, partition={}, offset={}", 
                topic, partition, offset);
        });
    
    return processOrderAsync(order)
        .thenCompose(v -> message.ack());
}
----

*Common Metadata Types by Connector*:

* *Kafka*: Topic, partition, offset, timestamp, key, headers
* *AMQP*: Exchange, routing key, content type, correlation ID
* *JMS*: Destination, delivery mode, priority, correlation ID
* *Custom*: Application-specific metadata attached via `Message.of(payload, metadata)`

Metadata access requires using the `Message<T>` wrapper rather than extracting just the payload.

=== Producing Messages with @Outgoing

The `@Outgoing` annotation marks a method as a producer of messages to a specified channel. Messages produced to the channel can be consumed by other methods annotated with `@Incoming` or sent to external brokers.

*Simple Value Production*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Outgoing;

@Outgoing("prices")
public double generatePrice() {
    return computeCurrentPrice();
}
----

This method is invoked repeatedly, and each returned value is sent as a message to the "prices" channel.

*Stream Production with Multi*:

[source,java]
----
import io.smallrye.mutiny.Multi;
import java.time.Duration;

@Outgoing("heartbeat")
public Multi<String> generateHeartbeat() {
    return Multi.createFrom().ticks().every(Duration.ofSeconds(5))
        .map(tick -> "alive");
}
----

This creates an infinite stream that emits "alive" every 5 seconds.

*Periodic Production with Timestamps*:

[source,java]
----
import java.time.Instant;

@Outgoing("ticks")
public Multi<Instant> generateTicks() {
    return Multi.createFrom().ticks().every(Duration.ofSeconds(1))
        .map(tick -> Instant.now());
}
----

*Producing Messages with Metadata*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Message;
import io.smallrye.reactive.messaging.kafka.api.OutgoingKafkaRecordMetadata;

@Outgoing("events")
public Message<Event> produceEvent(Event event) {
    OutgoingKafkaRecordMetadata metadata = OutgoingKafkaRecordMetadata.builder()
        .withKey(event.getAggregateId())
        .withTopic("events-topic")
        .build();
    
    return Message.of(event).addMetadata(metadata);
}
----

=== Message Processing Patterns

MicroProfile Reactive Messaging supports flexible transformation patterns by combining `@Incoming` and `@Outgoing` annotations:

*One-to-One Transformation (Mapping)*:

[source,java]
----
@Incoming("payment-requests")
@Outgoing("paid-orders")
public Order markOrderAsPaid(Order order) {
    order.setStatus(OrderStatus.PAID);
    return order;
}
----

Each incoming payment request produces exactly one outgoing order message with the updated payment status.

*One-to-Many Transformation (Splitting)*:

[source,java]
----
@Incoming("orders")
@Outgoing("order-items")
public Multi<OrderItem> splitOrder(Order order) {
    return Multi.createFrom().iterable(order.getItems());
}
----

Each incoming order is split into multiple order items, with each item sent as a separate message downstream.

*Many-to-One Transformation (Aggregation)*:

[source,java]
----
@Incoming("order-items")
@Outgoing("order-totals")
public Multi<OrderSummary> aggregateOrderItems(Multi<OrderItem> items) {
    return items
        .group().intoLists().of(5)
        .map(batch -> {
            BigDecimal total = batch.stream()
                .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
                .reduce(BigDecimal.ZERO, BigDecimal::add);

            return new OrderSummary(batch.get(0).getOrderId(), total, batch.size());
        });
}
----

Multiple `OrderItem` messages are aggregated into a single order summary, such as a total value and item count for downstream checkout or payment processing.

*Filtering Messages*:

[source,java]
----
@Incoming("all-events")
@Outgoing("important-events")
public Multi<Event> filterEvents(Multi<Event> events) {
    return events.filter(event -> event.getPriority() > 5);
}
----

Only events meeting the filter criteria are passed downstream. Low-priority events are discarded and automatically acknowledged.

*Asynchronous Enrichment*:

[source,java]
----
@Incoming("user-ids")
@Outgoing("users")
public CompletionStage<User> enrichUser(String userId) {
    return userRepository.findById(userId);
}
----

Each user ID triggers an asynchronous database lookup, and the enriched `User` object is sent downstream when the lookup completes.

*Conditional Routing (Dropping Messages)*:

[source,java]
----
@Incoming("transactions")
@Outgoing("high-value-transactions")
public Message<Transaction> routeHighValue(Message<Transaction> message) {
    Transaction tx = message.getPayload();
    
    if (tx.getAmount() > 10000) {
        return message;  // Forward high-value transactions
    } else {
        message.ack();  // Acknowledge but don't forward
        return null;    // Null signals no output
    }
}
----

*Stream Transformation Pipeline*:

[source,java]
----
@Incoming("raw-data")
@Outgoing("processed-data")
public Multi<ProcessedData> processStream(Multi<RawData> stream) {
    return stream
        .map(this::normalize)          // Transform
        .filter(this::isValid)          // Filter
        .onItem().transformToUniAndConcatenate(this::enrich)  // Async enrich
        .map(this::aggregate);          // Final transformation
}
----

These patterns can be combined to build sophisticated message processing pipelines declaratively, with the framework handling threading, backpressure, and error propagation automatically.

*Guidelines*:

* Use simple payload signatures for straightforward transformations
* Use `Multi<T>` when working with streams or requiring backpressure control
* Use `CompletionStage<T>` for async operations like database lookups or REST calls
* Use `Message<T>` when you need manual acknowledgment control or metadata access
* Return `null` from `@Outgoing` methods to drop messages without forwarding

== Programmatic Messaging with @Channel and @Emitter

Once the declarative model and the supported method signatures are clear, the imperative APIs become much easier to place in context. In the existing store application, this is the point where `OrderResource` can publish an order event after the order is accepted instead of synchronously coupling itself to downstream payment or shipment logic. Use `@Channel` when you need direct access to a stream, and use `Emitter` when imperative code such as REST endpoints needs to publish messages.

=== Using @Channel to Access Channels

The `@Channel` annotation allows developers to inject a channel directly into a bean, providing programmatic access to the message stream. This is useful for observing message flows or integrating reactive streams with imperative code:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Channel;
import io.smallrye.mutiny.Multi;
import jakarta.inject.Inject;

@ApplicationScoped
public class OrderService {
    
    @Inject
    @Channel("orders")
    Multi<Order> orders;
    
    public void displayOrders() {
        orders.subscribe().with(
            order -> System.out.println("Order: " + order.getId())
        );
    }
}
----

The injected type depends on the stream's cardinality:

* `Multi<T>` for streams of multiple elements
* `Publisher<T>` for Reactive Streams Publisher
* `PublisherBuilder<T>` for Reactive Streams operators

=== Using @Emitter for Programmatic Message Emission

An `Emitter` provides a way to programmatically send messages to a channel from imperative code, such as REST endpoints, scheduled tasks, or event handlers. This bridges imperative and reactive programming models:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Emitter;
import org.eclipse.microprofile.reactive.messaging.Channel;
import jakarta.inject.Inject;

@ApplicationScoped
public class OrderController {
    
    @Inject
    @Channel("orders")
    Emitter<Order> orderEmitter;
    public void createOrder(Order order) {
        orderEmitter.send(order);
    }
}
----

*Emitter Features*:

* *Payload Sending*: `send(T payload)` sends a payload and returns `CompletionStage<Void>`. The `CompletionStage` completes when the emitted message is acknowledged. If processing fails, the `CompletionStage` completes exceptionally.
* *Message Sending*: `send(Message<T> message)` sends a `Message` object and returns `void`. Use this form when you need custom acknowledgment or negative-acknowledgment functions.
* *Backpressure Aware*: Emitters respect backpressure from downstream consumers, preventing overwhelming slow processors. See the "Managing Backpressure" section for detailed backpressure control with `@OnOverflow`.

*Emitter with Payload Acknowledgment*:

[source,java]
----
@Inject
@Channel("orders")
Emitter<Order> orderEmitter;

public CompletionStage<Void> createOrderAsync(Order order) {
    return orderEmitter.send(order)
        .thenAccept(v -> log.info("Order sent successfully"))
        .exceptionally(throwable -> {
            log.error("Failed to send order", throwable);
            return null;
        });
}
----

*Emitter with Message and Custom Acknowledgment*:

[source,java]
----
@Inject
@Channel("orders")
Emitter<Order> orderEmitter;

public void createOrderWithAck(Order order) {
    orderEmitter.send(Message.of(order,
        () -> {
            log.info("Order acknowledged: " + order.getId());
            return CompletableFuture.completedFuture(null);
        },
        throwable -> {
            log.error("Order nacked: " + order.getId(), throwable);
            return CompletableFuture.completedFuture(null);
        }
    ));
}
----

*Use Cases for @Emitter*:

* *REST Endpoints*: Converting synchronous REST requests into messages for asynchronous processing.
* *Scheduled Tasks*: Periodically generating messages based on `@Scheduled` timers or cron expressions.
* *External Integrations*: Forwarding events from external systems (webhooks, file watchers) to message channels.
* *Event Sourcing*: Publishing domain events to event stores or Kafka topics after persisting them.
* *Testing*: Injecting test messages into channels programmatically during integration tests.

*Example: REST to Messaging Bridge*:

[source,java]
----
import jakarta.ws.rs.*;
import jakarta.ws.rs.core.MediaType;
import jakarta.ws.rs.core.Response;

@Path("/orders")
@ApplicationScoped
public class OrderResource {
    
    @Inject
    @Channel("orders")
    Emitter<Order> orderEmitter;
    
    @POST
    @Consumes(MediaType.APPLICATION_JSON)
    public Response createOrder(Order order) {
        orderEmitter.send(order);
        return Response.accepted().build();
    }
}
----

This pattern decouples the REST layer from the processing layer. It enables asynchronous processing and better scalability. The REST endpoint returns immediately while the order processing happens asynchronously in a separate messaging pipeline.

For fine-grained control over backpressure behavior when using emitters, see the "Controlling Backpressure with @OnOverflow" sub-section within "Managing Backpressure" section.

== Managing Backpressure in Reactive Messaging

Backpressure is one of the most critical challenges in reactive systems. It occurs when message producers generate data faster than consumers can process it. Without proper handling, this mismatch can lead to memory exhaustion, dropped messages, or system failures.

=== Understanding Backpressure

In a messaging system, backpressure scenarios arise from:

* *Fast Producer, Slow Consumer*: A Kafka topic receives thousands of messages per second, but the consumer can only process 100 per second due to database write latency.
* *Bursty Traffic*: Periodic spikes in message volume (for example, end-of-month batch jobs, flash sales) overwhelm downstream processors.
* *Resource Constraints*: Limited database connections, network bandwidth, or CPU capacity become bottlenecks during message processing.

*Traditional Approaches and Their Problems*:

1. *Unbounded Buffers*: Queue all messages in memory without limits, leading to `OutOfMemoryError` when the queue grows faster than it drains.
2. *Dropping Messages*: Silently discard excess messages to prevent buffer overflow, causing data loss and inconsistent state.
3. *Blocking Producers*: Block producers until consumers catch up, reducing overall throughput and potentially causing cascading failures across distributed systems.

Reactive Streams' request-based flow control provides a superior solution: consumers explicitly signal how many elements they can handle through the `Subscription.request(n)` method, and producers respect these demand signals by never sending more than requested.

=== Automatic Backpressure Management

MicroProfile Reactive Messaging automatically manages backpressure through the Reactive Streams protocol. The framework handles demand signaling and flow control behind the scenes based on your method signatures.

*Sequential Processing Pattern*:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Incoming;
import org.eclipse.microprofile.reactive.messaging.Outgoing;

@Incoming("data")
@Outgoing("processed-data")
public String process(String data) {
    return transform(data);
}
----

The framework automatically:

1. Requests one element from the upstream channel
2. Waits for the `process` method to complete
3. Sends the result downstream
4. Requests the next element only after the current one is fully processed

This sequential processing naturally provides backpressure: the upstream producer will not send more data until the current item is processed. If `transform(data)` takes 100ms, the consumer processes at most 10 messages per second, and the producer adapts to this rate.

*Asynchronous Processing with Backpressure*:

[source,java]
----
import java.util.concurrent.CompletionStage;
import java.util.concurrent.CompletableFuture;

@Incoming("tasks")
@Outgoing("results")
public CompletionStage<Result> processAsync(Task task) {
    return CompletableFuture.supplyAsync(() -> 
        expensiveOperation(task)
    );
}
----

The framework waits for each `CompletionStage` to complete before requesting the next message. This ensures backpressure is maintained even when processing happens asynchronously on different threads or involves I/O operations.

*Stream Processing with Reactive Types*:

[source,java]
----
import io.smallrye.mutiny.Multi;

@Incoming("events")
@Outgoing("filtered-events")
public Multi<Event> filterStream(Multi<Event> events) {
    return events
        .filter(event -> event.isImportant())
        .map(this::enrich);
}
----

When using reactive types like `Multi`, backpressure propagates automatically through the entire chain of operators. If a downstream subscriber can only handle 10 items, the `filter` and `map` operators request only what's needed from upstream.

=== Configuring Backpressure Behavior

While automatic backpressure works well for many scenarios, you can configure specific behaviors for advanced requirements:

*Buffering Strategy*:

Configure buffer sizes for incoming channels to smooth out temporary speed mismatches:

[source,properties]
----
# Allow producer to send up to 256 messages ahead
mp.messaging.incoming.data.buffer-size=256
----

This creates a buffer of 256 messages between the connector and your consumer method. The producer can send messages at its natural rate while the consumer processes at its own pace, as long as the buffer doesn't fill completely.

*Choosing Buffer Size*:

* *Small buffers (10-50)*: Tight backpressure control, minimal memory usage, but frequent demand signals
* *Medium buffers (100-500)*: Good balance for most scenarios with occasional bursts
* *Large buffers (1000+)*: Tolerate significant bursts, but consume more memory and delay backpressure signals

*Concurrent Processing*:

Process multiple messages concurrently while maintaining overall backpressure control:

[source,java]
----
import io.smallrye.mutiny.Multi;
import io.smallrye.mutiny.Uni;

@Incoming("tasks")
@Outgoing("results")
public Multi<Result> processConcurrent(Multi<Task> tasks) {
    return tasks
        .onItem().transformToUniAndMerge(task ->
            Uni.createFrom().completionStage(() ->
                CompletableFuture.supplyAsync(() -> process(task))
            ),
            3  // Process up to 3 messages concurrently
        );
}
----

The concurrency limit (3 in this example) ensures that no more than 3 messages are processed simultaneously. This provides controlled parallelism for I/O-bound operations while preventing resource exhaustion.

*Batching for Throughput*:

Process messages in batches to improve throughput for operations that benefit from bulk processing:

[source,java]
----
import io.smallrye.mutiny.Multi;
import java.time.Duration;

@Incoming("events")
@Outgoing("batched-events")
public Multi<List<Event>> batch(Multi<Event> events) {
    return events
        .group().intoLists().of(100, Duration.ofSeconds(5));
}
----

This batches up to 100 events or waits up to 5 seconds, whichever comes first. Batching is particularly effective for:

* Bulk database inserts
* Batch API calls to external services
* Aggregated analytics computations

*Rate Limiting*:

Control the maximum processing rate to prevent overwhelming downstream systems:

[source,java]
----
import io.smallrye.mutiny.Multi;
import io.smallrye.mutiny.Uni;
import java.time.Duration;

@Incoming("high-volume")
@Outgoing("rate-limited")
public Multi<Data> rateLimit(Multi<Data> stream) {
    return stream
        .group().intoLists().of(10)                    // Batch 10 items
        .onItem().call(batch ->                         // Pause between batches
            Uni.createFrom().nullItem()
                .onItem().delayIt().by(Duration.ofMillis(100))
        )
        .onItem().transformToMultiAndConcatenate(Multi::createFrom().iterable);
}
----

This processes at most 100 messages per second (10 messages every 100ms), protecting downstream services from overload.

=== Controlling Backpressure with @OnOverflow

When using `@Emitter` to programmatically send messages to channels, the `@OnOverflow` annotation controls what happens when messages are emitted faster than downstream consumers can process them. Without this annotation, the default strategy is `OnOverflow.Strategy.BUFFER` with a buffer size of 128 elements (or the value of the `mp.messaging.emitter.default-buffer-size` configuration property).

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Channel;
import org.eclipse.microprofile.reactive.messaging.Emitter;
import org.eclipse.microprofile.reactive.messaging.OnOverflow;
import jakarta.inject.Inject;

@Inject
@Channel("orders")
@OnOverflow(value = OnOverflow.Strategy.BUFFER, bufferSize = 300)
Emitter<Order> orderEmitter;
----

*Available Overflow Strategies*:

[cols="1,3"]
|===
|Strategy |Behavior

|`BUFFER`
|Buffer messages up to `bufferSize` elements. If `bufferSize` is not set, the `mp.messaging.emitter.default-buffer-size` property is used (default: 128). Throws `java.lang.IllegalStateException` from `send()` when the buffer is full.

|`UNBOUNDED_BUFFER`
|Use an unbounded buffer. Messages accumulate in memory without limit. The application may run out of memory if messages accumulate faster than they are consumed. Use only when you can guarantee consumption keeps pace.

|`THROW_EXCEPTION`
|Throw `IllegalStateException` from `send()` immediately if the downstream cannot keep up. No buffering occurs.

|`DROP`
|Drop the most recently emitted message when the downstream cannot keep up. Previously buffered messages are retained. This strategy prioritizes older messages.

|`FAIL`
|Propagate a failure signal downstream when backpressure is detected. The emitter stops accepting new messages and the stream terminates.

|`LATEST`
|Keep only the most recent message, discarding all previously buffered messages when the downstream cannot keep up. This strategy prioritizes the newest data.

|`NONE`
|Ignore backpressure signals completely. The downstream consumer must implement its own buffering strategy. Use with caution as this can lead to unbounded memory growth.
|===

*Choosing the Right Strategy*:

* *`BUFFER`*: Best for most scenarios where occasional slowdowns are acceptable and you want to preserve all messages. Good for business transactions, orders, or events that must not be lost.
* *`DROP` or `LATEST`*: Appropriate for high-frequency sensor data, metrics, or real-time dashboards where missing some values is acceptable and recent data is more valuable than complete history.
* *`FAIL`*: Use when any backpressure indicates a system problem that should halt processing and trigger alerts. Appropriate for critical systems where degraded performance should stop operations.
* *`THROW_EXCEPTION`*: Provides immediate feedback to the caller that the system is overloaded. Useful in REST endpoints to return HTTP 503 Service Unavailable when the system cannot accept more requests.
* *`UNBOUNDED_BUFFER`*: Rarely recommended. Only use in controlled environments with guaranteed bounded input (for example, processing a fixed-size file).

*Strategy Examples by Use Case*:

[source,java]
----
// High-frequency sensor data where latest value matters most
@Inject
@Channel("temperature-readings")
@OnOverflow(OnOverflow.Strategy.LATEST)
Emitter<Temperature> temperatureEmitter;

// Critical transactions that must all be processed
@Inject
@Channel("payments")
@OnOverflow(value = OnOverflow.Strategy.BUFFER, bufferSize = 1000)
Emitter<Payment> paymentEmitter;

// REST endpoint that should fail fast when overloaded
@Inject
@Channel("api-requests")
@OnOverflow(OnOverflow.Strategy.THROW_EXCEPTION)
Emitter<ApiRequest> requestEmitter;

@POST
@Path("/requests")
public Response handleRequest(ApiRequest request) {
    try {
        requestEmitter.send(request);
        return Response.accepted().build();
    } catch (IllegalStateException e) {
        return Response.status(503).entity("System overloaded").build();
    }
}
----

When the `@OnOverflow` annotation is absent, the `BUFFER` strategy with default buffer size applies automatically. This provides a sensible default for most applications while allowing fine-grained control when needed.

=== Backpressure Best Practices

*1. Match Buffer Sizes to Message Characteristics*:

* Small, frequent messages: Larger buffers (500-1000)
* Large, infrequent messages: Smaller buffers (10-50)
* Variable sizes: Monitor memory and adjust dynamically

*2. Use Concurrency for I/O-Bound Operations*:

[source,java]

// Good: Concurrent HTTP calls
@Incoming("user-ids")
public Multi<User> fetchUsers(Multi<String> userIds) {
    return userIds.onItem().transformToUniAndMerge(

----

*3. Batch for Efficiency, Not to Bypass Backpressure*:

Batching should improve throughput for operations that benefit from bulk processing, not hide backpressure problems.

*4. Choose @OnOverflow Strategies Based on Data Criticality*:

* Critical data (payments, orders): Use `BUFFER` or `THROW_EXCEPTION`
* Transient data (metrics, sensors): Use `DROP` or `LATEST`
* Monitor dropped messages with metrics when using lossy strategies

*5. Monitor Backpressure Metrics*:

Use MicroProfile Metrics to track:

* Buffer fill rates
* Processing latency
* Message drop counts (if using DROP strategy)
* Backpressure events

*6. Test Under Load*:

Understanding and properly configuring backpressure ensures your reactive messaging applications remain stable and responsive under varying load conditions. The framework's automatic handling covers most scenarios, while the configuration options and `@OnOverflow` strategies provide fine-grained control when needed.

== Message Acknowledgment and Error Handling

Reliable message processing requires proper acknowledgment and error handling strategies. MicroProfile Reactive Messaging provides flexible mechanisms to ensure messages are processed correctly and failures are handled appropriately.

=== Understanding Message Acknowledgment

Acknowledgment (ack) is the signal sent to the messaging system confirming that a message has been successfully processed. Negative acknowledgment (nack) signals that processing failed and the message should be retried or sent to a dead letter queue.

Acknowledgment behavior directly affects message delivery guarantees:

* *At-most-once*: Messages are acknowledged before processing. If processing fails, the message is lost.
* *At-least-once*: Messages are acknowledged after successful processing. If processing fails, the message is redelivered, potentially causing duplicates.
* *Exactly-once*: Requires idempotent processing or transactional coordination (implementation-specific).

=== Acknowledgment Strategies

MicroProfile Reactive Messaging supports several acknowledgment strategies configured via the `@Acknowledgment` annotation:

*1. POST_PROCESSING (Default)*:

Acknowledge the message after the method completes successfully:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Incoming;
import org.eclipse.microprofile.reactive.messaging.Acknowledgment;

@Incoming("orders")
@Acknowledgment(Acknowledgment.Strategy.POST_PROCESSING)
public void processOrder(Order order) {
    orderService.save(order);
    // Message acknowledged after method returns successfully
    // If an exception is thrown, message is NOT acknowledged
}
----

* *Use when*: You need at-least-once delivery guarantees and want automatic acknowledgment.
* *Behavior*: If the method throws an exception, the message is not acknowledged and will be redelivered (subject to connector retry policies).
* *Default for*: Methods with `void` return type or methods returning simple types.

*Async POST_PROCESSING*:

[source,java]
----
@Incoming("orders")
@Acknowledgment(Acknowledgment.Strategy.POST_PROCESSING)
public CompletionStage<Void> processOrder(Order order) {
    return orderService.saveAsync(order);
    // Message acknowledged when CompletionStage completes successfully
    // If CompletionStage completes exceptionally, message is NOT acknowledged
}
----

*2. PRE_PROCESSING*:

Acknowledge the message before processing begins:

[source,java]
----
@Incoming("metrics")
@Acknowledgment(Acknowledgment.Strategy.PRE_PROCESSING)
public void recordMetric(Metric metric) {
    // Message already acknowledged before this line executes
    metricsRepository.save(metric);
    // If this fails, message is already acknowledged and won't be redelivered
}
----

* *Use when*: Processing is idempotent, message loss is acceptable, or you want at-most-once semantics.
* *Risk*: If processing fails, the message is already acknowledged and will not be redelivered.
* *Best for*: High-throughput scenarios where occasional loss is acceptable (metrics, analytics, sensor data).

*3. MANUAL*:

Manually control when and how to acknowledge:

[source,java]
----
import org.eclipse.microprofile.reactive.messaging.Message;

@Incoming("payments")
@Acknowledgment(Acknowledgment.Strategy.MANUAL)
public CompletionStage<Void> processPayment(Message<Payment> message) {
    Payment payment = message.getPayload();
    
    return paymentService.process(payment)
        .thenCompose(result -> {
            log.info("Payment processed successfully: {}", payment.getId());
            return message.ack();  // Explicit acknowledgment
        })
        .exceptionally(throwable -> {
            log.error("Payment processing failed: {}", payment.getId(), throwable);
            return message.nack(throwable);  // Negative acknowledgment
        });
}
----

* *Use when*: You need fine-grained control over acknowledgment timing or conditional acknowledgment logic.
* *Methods*: 
  - `message.ack()` returns `CompletionStage<Void>` for acknowledgment
  - `message.nack(throwable)` returns `CompletionStage<Void>` for negative acknowledgment with reason
  - `message.nack(throwable, metadata)` allows passing additional metadata about the failure

*Conditional Acknowledgment*:

[source,java]
----
@Incoming("events")
@Acknowledgment(Acknowledgment.Strategy.MANUAL)
public CompletionStage<Void> handleEvent(Message<Event> message) {
    Event event = message.getPayload();
    
    return validateEvent(event)
        .thenCompose(isValid -> {
            if (isValid) {
                return processEvent(event)
                    .thenCompose(v -> message.ack());
            } else {
                // Invalid event - acknowledge to skip it (don't retry)
                log.warn("Skipping invalid event: {}", event.getId());
                return message.ack();
            }
        })
        .exceptionally(throwable -> {
            // Unexpected error - nack to retry
            log.error("Event processing error: {}", event.getId(), throwable);
            return message.nack(throwable);
        });
}
----

*4. NONE*:

No automatic acknowledgment:

[source,java]
----
@Incoming("logs")
@Acknowledgment(Acknowledgment.Strategy.NONE)
public void logMessage(String message) {
    logger.info(message);
    // No acknowledgment occurs - connector manages it
}
----

* *Use when*: The connector or external system handles acknowledgment, or acknowledgment is not supported.
* *Rare use case*: Most applications should use one of the other strategies.

=== Default Acknowledgment Behavior

If no `@Acknowledgment` annotation is present, the default strategy depends on the method signature:

[cols="2,1"]
|===
|Method Signature |Default Strategy

|`void method(T payload)`
|POST_PROCESSING

|`CompletionStage<Void> method(T payload)`
|POST_PROCESSING

|`void method(Message<T> message)`
|MANUAL

|`CompletionStage<Void> method(Message<T> message)`
|MANUAL

|`Multi<R> method(Multi<T> stream)`
|POST_PROCESSING (per item)
|===

When using `Message<T>` wrapper, MANUAL is the default because it signals you want explicit control over acknowledgment.

=== Error Handling Strategies

When message processing fails, MicroProfile Reactive Messaging provides several strategies to handle the failure. These are configured per incoming channel:

*1. Failure Strategy: fail (Default)*:

Stop processing and propagate the failure:

[source,properties]
----
mp.messaging.incoming.orders.failure-strategy=fail
----

* *Behavior*: When a message fails, the entire channel stops processing. The application container may attempt to restart the channel based on the connector implementation.
* *Use when*: Any failure indicates a critical problem that requires investigation and manual intervention.

*2. Failure Strategy: ignore*:

Log the failure and continue processing subsequent messages:

[source,properties]
----
mp.messaging.incoming.metrics.failure-strategy=ignore
----

* *Behavior*: Failed messages are logged and discarded. Processing continues with the next message.
* *Use when*: Individual message failures should not stop the stream (for example, best-effort metrics collection).
* *Risk*: Failed messages are permanently lost.

*3. Failure Strategy: dead-letter-queue*:

Send failed messages to a separate channel or topic for later analysis:

[source,properties]
----
# Kafka connector example
mp.messaging.incoming.orders.failure-strategy=dead-letter-queue
mp.messaging.incoming.orders.dead-letter-queue.topic=orders-dlq
mp.messaging.incoming.orders.dead-letter-queue.key.serializer=org.apache.kafka.common.serialization.StringSerializer
mp.messaging.incoming.orders.dead-letter-queue.value.serializer=org.apache.kafka.common.serialization.StringSerializer
----

* *Behavior*: Failed messages are sent to the configured dead letter queue topic. Original message processing continues with the next message.
* *Use when*: You want to preserve failed messages for later analysis and reprocessing without blocking the main stream.
* *Requires*: Connector support (available in Kafka, AMQP connectors).

*4. Custom Failure Handling with MANUAL Acknowledgment*:

Implement custom logic for different failure scenarios:

[source,java]
----
@Incoming("orders")
@Acknowledgment(Acknowledgment.Strategy.MANUAL)
public CompletionStage<Void> processOrder(Message<Order> message) {
    Order order = message.getPayload();
    
    return orderService.process(order)
        .thenCompose(result -> message.ack())
        .exceptionally(throwable -> {
            if (throwable instanceof ValidationException) {
                // Business validation failure - skip this message
                log.warn("Invalid order {}: {}", order.getId(), throwable.getMessage());
                auditLog.recordInvalidOrder(order, throwable);
                return message.ack();  // Acknowledge to prevent redelivery
                
            } else if (throwable instanceof TemporaryException) {
                // Transient failure - retry via nack
                log.warn("Temporary failure for order {}, will retry", order.getId());
                return message.nack(throwable);
                
            } else {
                // Unexpected error - nack and alert
                log.error("Unexpected error processing order {}", order.getId(), throwable);
                alertService.sendAlert("Order processing failed", throwable);
                return message.nack(throwable);
            }
        });
}
----

This pattern allows different handling for:

* *Validation errors*: Acknowledged and logged (no retry)
* *Transient errors*: Negative acknowledged for retry
* *Unexpected errors*: Negative acknowledged and alerts sent

=== Integration with MicroProfile Fault Tolerance

Combine message processing with MicroProfile Fault Tolerance for robust error handling:

*Retry with Circuit Breaker*:

[source,java]
----
import org.eclipse.microprofile.faulttolerance.CircuitBreaker;
import org.eclipse.microprofile.faulttolerance.Retry;
import org.eclipse.microprofile.faulttolerance.Timeout;
import java.time.temporal.ChronoUnit;

@ApplicationScoped
public class OrderProcessor {
    
    @Inject
    ExternalInventoryService inventoryService;
    
    @Incoming("orders")
    @Acknowledgment(Acknowledgment.Strategy.POST_PROCESSING)
    @Retry(maxRetries = 3, delay = 1, delayUnit = ChronoUnit.SECONDS)
    @Timeout(value = 5, unit = ChronoUnit.SECONDS)
    @CircuitBreaker(
        requestVolumeThreshold = 10,
        failureRatio = 0.5,
        delay = 10,
        delayUnit = ChronoUnit.SECONDS
    )
    public CompletionStage<Void> processOrder(Order order) {
        return inventoryService.checkAvailability(order)
            .thenCompose(available -> {
                if (available) {
                    return inventoryService.reserveItems(order);
                } else {
                    throw new OutOfStockException();
                }
            });
    }
}
----

This configuration:

* *@Retry*: Retries up to 3 times with 1-second delays for transient failures
* *@Timeout*: Fails if external service call exceeds 5 seconds
* *@CircuitBreaker*: Opens circuit if 50% of last 10 requests fail, preventing cascade failures

*Fallback for Degraded Service*:

[source,java]
----
import org.eclipse.microprofile.faulttolerance.Fallback;

@Incoming("notifications")
@CircuitBreaker(requestVolumeThreshold = 5, failureRatio = 0.4)
@Fallback(fallbackMethod = "fallbackNotification")
public CompletionStage<Void> sendNotification(Notification notification) {
    return emailService.send(notification);
}

public CompletionStage<Void> fallbackNotification(Notification notification) {
    log.warn("Email service unavailable, queuing notification for later");
    return notificationQueue.enqueue(notification);
}
----

*Bulkhead for Resource Isolation*:

[source,java]
----
import org.eclipse.microprofile.faulttolerance.Bulkhead;

@Incoming("file-uploads")
@Bulkhead(value = 5, waitingTaskQueue = 10)
public CompletionStage<Void> processUpload(FileUpload upload) {
    return fileProcessor.process(upload);
}
----

Limits concurrent file processing to 5, with up to 10 waiting in queue, preventing resource exhaustion.

=== Best Practices for Acknowledgment and Error Handling

*1. Choose Acknowledgment Strategy Based on Delivery Guarantees*:

* *At-least-once needed*: Use POST_PROCESSING (default)
* *At-most-once acceptable*: Use PRE_PROCESSING for high throughput
* *Complex logic required*: Use MANUAL for fine control
* *Idempotent processing*: Design for at-least-once, handle duplicates gracefully

== Reactive Stream Operations

Reactive stream operations provide powerful tools for transforming, filtering, and combining message streams. MicroProfile Reactive Messaging implementations support reactive operators through libraries like Mutiny, enabling functional-style stream processing with backpressure management.

=== Transformation Operators

*Map - Transform Elements*:

Transform each element in the stream:

[source,java]
----
import io.smallrye.mutiny.Multi;

@Incoming("prices-cents")
@Outgoing("prices-dollars")
public Multi<Double> convertToDollars(Multi<Integer> prices) {
    return prices.map(cents -> cents / 100.0);
}
----

*FlatMap - One-to-Many Transformation*:

Transform each element into a stream of elements:

[source,java]
----
@Incoming("orders")
@Outgoing("items")
public Multi<Item> extractItems(Multi<Order> orders) {
    return orders
        .onItem().transformToMultiAndConcatenate(order ->
            Multi.createFrom().iterable(order.getItems())
        );
}
----

Each order is expanded into multiple items, which are then flattened into a single stream.

*OnItem.transform - Asynchronous Transformation*:

Transform elements asynchronously with external service calls:

[source,java]
----
@Incoming("user-ids")
@Outgoing("users")
public Multi<User> enrichUsers(Multi<String> userIds) {
    return userIds
        .onItem().transformToUniAndConcatenate(userId ->
            userService.fetchUser(userId)  // Returns Uni<User>
        );
}
----

=== Filtering Operators

*Filter - Select Elements*:

Select elements based on a predicate:

[source,java]
----
@Incoming("all-orders")
@Outgoing("large-orders")
public Multi<Order> filterLargeOrders(Multi<Order> orders) {
    return orders.filter(order -> order.getTotal() > 1000.0);
}
----

Only orders with totals exceeding $1000 are passed downstream.

*Distinct - Remove Duplicates*:

Remove duplicate elements from the stream:

[source,java]
----
@Incoming("events")
@Outgoing("unique-events")
public Multi<Event> removeDuplicates(Multi<Event> events) {

        .select().distinct(Event::getId);
}
----

*Skip and Take - Limit Stream Elements*:

Skip the first N elements:

[source,java]
----
@Incoming("events")
@Outgoing("skip-header")
public Multi<Event> skipHeader(Multi<Event> events) {

}
----

Take only the first N elements:

[source,java]
----
@Incoming("events")
@Outgoing("first-ten")
public Multi<Event> takeFirst(Multi<Event> events) {
    return events
        .select().first(10);  // Take only first 10 events
}
----

=== Observability Operators

*Invoke - Side Effects*:

Perform side effects without modifying the stream:

[source,java]
----
@Incoming("orders")
@Outgoing("processed-orders")
public Multi<Order> logOrders(Multi<Order> orders) {
    return orders
        .invoke(order -> log.info("Processing order: {}", order.getId()))
        .invoke(order -> metrics.incrementOrderCount());
}
----

The `invoke` operator is useful for logging, metrics, or debugging without affecting the data flow.

*Call - Asynchronous Side Effects*:

Perform asynchronous side effects that return `Uni<Void>`:

[source,java]
----
@Incoming("orders")
@Outgoing("orders-logged")
public Multi<Order> auditOrders(Multi<Order> orders) {
    return orders
        .onItem().call(order ->
            auditService.logOrder(order)  // Returns Uni<Void>
        );
}
----

The stream waits for each audit operation to complete before proceeding, maintaining order and backpressure.

=== Aggregation Operators

*Group and Batch*:

Collect elements into batches:

[source,java]
----
@Incoming("events")
@Outgoing("batched-events")
public Multi<List<Event>> batchEvents(Multi<Event> events) {
    return events
        .group().intoLists().of(100);  // Batch 100 events at a time
}
----

With time-based batching:

[source,java]
----

@Incoming("sensor-readings")
@Outgoing("batched-readings")

    return readings
        .group().intoLists().of(50, Duration.ofSeconds(5));

----

*Collect - Terminal Aggregation*:

Aggregate all elements into a single result:

[source,java]
----
@Incoming("values")
public Uni<Double> calculateAverage(Multi<Double> values) {
    return values
        .collect().asList()
        .onItem().transform(list ->
            list.stream()
                .mapToDouble(Double::doubleValue)
                .average()
                .orElse(0.0)
        );
}
----

Note: This consumes the entire stream and returns a `Uni`, not suitable for infinite streams.

*Scan - Running Aggregation*:

Maintain running aggregates across elements:

[source,java]
----
@Incoming("transactions")
@Outgoing("running-balances")
public Multi<Balance> calculateRunningBalance(Multi<Transaction> transactions) {
    return transactions
        .onItem().scan(() -> new Balance(0.0), 
            (balance, transaction) -> {
                balance.add(transaction.getAmount());
                return balance;
            }
        );
}
----

Each emitted balance reflects all transactions processed so far.

=== Combining Streams

*Merge - Combine Multiple Streams*:

Merge multiple streams into one, emitting items as they arrive:

[source,java]
----
@ApplicationScoped
public class StreamCombiner {
    
    @Inject
    @Channel("events-source-1")
    Multi<Event> stream1;
    
    @Inject
    @Channel("events-source-2")
    Multi<Event> stream2;
    
    @Outgoing("merged-events")
    public Multi<Event> mergeStreams() {
        return Multi.createBy().merging()
            .streams(stream1, stream2);
    }
}
----

Items from both streams are interleaved based on their arrival time.

*Concatenate - Sequential Combination*:

Combine streams sequentially (exhaust first stream before starting second):

[source,java]
----
@Outgoing("concatenated-events")
public Multi<Event> concatenateStreams() {
    return Multi.createBy().concatenating()
        .streams(historicalEvents, liveEvents);
}
----

*Combine - Pair Elements*:

Combine corresponding elements from multiple streams:

[source,java]
----
@ApplicationScoped
public class PricingService {
    
    @Inject
    @Channel("orders")
    Multi<Order> orders;
    
    @Inject
    @Channel("prices")
    Multi<Price> prices;
    
    @Outgoing("orders-with-prices")
    public Multi<OrderWithPrice> combineStreams() {
        return Multi.createBy().combining()
            .streams(orders, prices)
            .using((order, price) -> new OrderWithPrice(order, price));
    }
}
----

Elements are paired based on their index in the stream. If streams have different lengths, the combined stream completes when the shorter stream ends.

Share a single stream with multiple subscribers:

*OnFailure.transform - Convert Exceptions*:

=== Rate Control Operators

*Throttle - Limit Emission Rate*:

Control the maximum rate of element emission:

[source,java]
----
@Incoming("high-frequency-events")
@Outgoing("throttled-events")
public Multi<Event> throttleEvents(Multi<Event> events) {
    return events
        .onItem().call(event ->
            Uni.createFrom().nullItem()
                .onItem().delayIt().by(Duration.ofMillis(100))
        );
}
----

This limits processing to approximately 10 events per second.

*Debounce - Ignore Rapid Emissions*:

Only emit items after a quiet period:

[source,java]
----
@Incoming("user-input")
@Outgoing("search-queries")
public Multi<String> debounceInput(Multi<String> input) {
    return input
        .group().intoLists().of(1, Duration.ofMillis(300))
        .onItem().transform(list -> list.get(list.size() - 1));
}
----

This is useful for search-as-you-type functionality, waiting 300ms of inactivity before processing.

=== Best Practices for Stream Operations

*1. Preserve Backpressure*:

Always use operators that maintain backpressure. Avoid collecting entire streams into memory unless necessary:

[source,java]
----
// Good: Maintains backpressure
return stream
    .map(this::transform)
    .filter(this::isValid);

// Bad: Breaks backpressure, loads everything into memory
return Multi.createFrom().iterable(
    stream.collect().asList().await().indefinitely()
);
----

*2. Use Appropriate Operators for Async Operations*:

Use `transformToUniAndConcatenate` or `transformToUniAndMerge` for async transformations:

[source,java]
----
// Sequential async processing (preserves order)
.onItem().transformToUniAndConcatenate(item -> asyncOperation(item))

// Concurrent async processing (may reorder)
.onItem().transformToUniAndMerge(item -> asyncOperation(item), 10)
----

*3. Handle Errors Appropriately*:

Add error handling to prevent stream termination:

[source,java]
----
return stream
    .onItem().transformToUniAndConcatenate(item ->
        processItem(item)
            .onFailure().recoverWithItem(defaultValue)
    );
----

*4. Log for Observability*:

Use `invoke` for logging and metrics without disrupting the stream:

[source,java]
----
return stream
    .invoke(item -> log.debug("Processing: {}", item))
    .map(this::transform)
    .invoke(result -> metrics.recordSuccess());
----

*5. Batch for Efficiency*:

Use batching for operations that benefit from bulk processing:

[source,java]
----
return stream
    .group().intoLists().of(100, Duration.ofSeconds(5))
    .onItem().transformToUniAndConcatenate(batch ->
        database.insertBatch(batch)
    );
----

Reactive stream operations enable building sophisticated message processing pipelines declaratively while maintaining backpressure and resource efficiency throughout the entire flow.

=== Using @Blocking for Blocking Operations

[NOTE]
====
The `@Blocking` annotation is not defined in the MicroProfile Reactive Messaging 3.0.1 specification. It is an extension provided by the SmallRye Reactive Messaging implementation (used by Quarkus and other runtimes). Other implementations may provide equivalent mechanisms through different annotations or configuration. Refer to your runtime's documentation for portability guidance.
====

By default, MicroProfile Reactive Messaging processes messages on event loop threads, which must not be blocked. For blocking operations such as database calls, file I/O, or synchronous HTTP requests, use `@Blocking` to offload processing to a worker thread pool:

*Basic @Blocking Usage*:

[source,java]
----
import io.smallrye.reactive.messaging.annotations.Blocking;
import jakarta.persistence.EntityManager;
import jakarta.inject.Inject;

@ApplicationScoped
public class UserUpdateProcessor {
    
    @Inject
    EntityManager entityManager;
    
    @Incoming("user-updates")
    @Blocking
    public void updateUser(User user) {
        // Blocking database operation - safe because of @Blocking
        entityManager.merge(user);
        entityManager.flush();
    }
}
----

*Advanced @Blocking Configuration*:

[source,java]
----
@Incoming("cpu-intensive-tasks")
@Blocking(value = "cpu-pool", ordered = false)
public Result processIntensiveTask(Task task) {
    // CPU-intensive computation
    return computeResult(task);
}
----

*@Blocking Parameters*:

* `value`: Custom worker pool name. If not specified, uses the default worker pool.
* `ordered`: Whether to preserve message order (default: `true`). Set to `false` for higher throughput when order doesn't matter.

*Configuring Custom Worker Pools*:

[source,properties]
----
# Default worker pool size
smallrye.messaging.worker.pool-size=10

# Custom pool configuration
smallrye.messaging.worker.cpu-pool.pool-size=20
smallrye.messaging.worker.cpu-pool.queue-size=1000
----

*When to Use @Blocking*:

* Database operations (JPA, JDBC queries, transactions)
* File system operations (reading, writing files)
* Synchronous HTTP client calls
* CPU-intensive computations
* Thread.sleep() or blocking synchronization

*When NOT to Use @Blocking*:

* Asynchronous operations returning `CompletionStage<T>` or `Uni<T>`
* Non-blocking I/O operations
* Simple, fast transformations (< 1ms)
* Operations already running on worker threads

*Performance Implications*:

[source,java]
----
// Without @Blocking - blocks event loop (BAD)
@Incoming("orders")
public void processOrder(Order order) {
    // This BLOCKS the event loop thread!
    entityManager.persist(order);  // Database I/O
}

// With @Blocking - uses worker thread (GOOD)

@ApplicationScoped

== Best Practices and Production Patterns

=== Graph Validation and Deployment Checks

When the application starts, the MicroProfile Reactive Messaging implementation constructs a directed graph connecting all `@Incoming` and `@Outgoing` methods, `Emitter` instances, `@Channel` injections, and connectors. The implementation validates this graph and throws a `DeploymentException` if any of the following invalid conditions exist:

*Invalid Channel Configurations*:

* A method annotated with `@Incoming` has no upstream channel
* A method annotated with `@Outgoing` has no downstream channel
* A method annotated with `@Incoming` has multiple upstream channels (fan-in not supported)
* A method annotated with `@Outgoing` has multiple downstream channels (fan-out requires multiple methods)
* An `Emitter` has no downstream channel
* An `Emitter` has multiple downstream channels
* An injected `@Channel` has no upstream channel
* An injected `@Channel` has multiple upstream channels

*Invalid Connector Configurations*:

* The application references a connector that is not available in the runtime
* An incoming connector has no downstream channel
* An incoming connector has multiple downstream channels
* An outgoing connector has no upstream channel
* An outgoing connector has multiple upstream channels

[IMPORTANT]
====
*Channel Uniqueness Constraint*

Each channel name may appear in at most **one** `@Incoming` annotation across the entire application. If two methods are annotated with `@Incoming("same-channel")`, the implementation treats this as multiple downstream consumers for a single channel, which violates the specification and causes a `DeploymentException` at startup.

The same constraint applies to `@Outgoing`: a given channel name may appear in at most **one** `@Outgoing` annotation.

This design enforces a clear, unambiguous message flow graph. To broadcast messages to multiple consumers, use an intermediate method that fans out to multiple channels:

[source,java]
----
@Incoming("source")
@Outgoing("dest1")
@Outgoing("dest2")  // ERROR: Multiple @Outgoing not allowed

// Instead, use explicit fan-out:
@Incoming("source")
public Multi<Message> fanOut(Message msg) {
    channel1Emitter.send(msg);
    channel2Emitter.send(msg);
    return Multi.createFrom().empty();
}

=== Implementing Idempotent Message Processing

Messages may be delivered more than once due to network issues, broker failures, or rebalancing. All message processing must be idempotent to handle duplicate deliveries correctly.

*Idempotency Key Pattern*:

[source,java]
----
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

@ApplicationScoped
public class IdempotentPaymentProcessor {
    
    private final Set<String> processedPayments = 
        ConcurrentHashMap.newKeySet();
    
    @Incoming("payments")
    @Blocking
    @Acknowledgment(Acknowledgment.Strategy.MANUAL)
    public CompletionStage<Void> processPayment(Message<Payment> message) {
        Payment payment = message.getPayload();
        String idempotencyKey = payment.getIdempotencyKey();
        
        // Check if already processed
        if (processedPayments.contains(idempotencyKey)) {
            log.info("Payment already processed: {}", idempotencyKey);
            return message.ack();
        }
        
        // Process payment
        return paymentService.process(payment)
            .thenCompose(result -> {
                processedPayments.add(idempotencyKey);
                return message.ack();
            })
            .exceptionally(throwable -> {
                log.error("Payment processing failed", throwable);
                return message.nack(throwable);
            });
    }
}
----

*Database-Based Idempotency with Unique Constraints*:

[source,java]
----
@ApplicationScoped
public class OrderProcessor {
    
    @Inject
    EntityManager entityManager;
    
    @Incoming("orders")
    @Blocking
    @Transactional
    public void processOrder(Order order) {
        try {
            // Attempt to insert with unique constraint on external_id
            ProcessedOrder processed = new ProcessedOrder();
            processed.setExternalId(order.getExternalId());
            processed.setOrderData(order);
            processed.setProcessedAt(Instant.now());
            
            entityManager.persist(processed);
            entityManager.flush();
            
            // If we get here, this is the first time we've seen this order
            performBusinessLogic(order);
            
        } catch (PersistenceException e) {
            if (isUniqueConstraintViolation(e)) {
                // Already processed - safe to ignore
                log.info("Order already processed: {}", order.getExternalId());
            } else {
                throw e;  // Unexpected database error
            }
        }
    }
    
    private boolean isUniqueConstraintViolation(PersistenceException e) {
        return e.getCause() instanceof ConstraintViolationException;
    }
}

== Data Streaming and Event Sourcing

Modern applications often need to handle continuous flows of data and maintain
comprehensive audit trails of state changes. Data streaming and event sourcing
are two complementary patterns that build on the core reactive messaging model
introduced earlier in this chapter.

=== Data Streaming

Data streaming involves processing a continuous flow of data as it arrives,
rather than in discrete batches. Unlike traditional batch processing, where
data is collected and processed at scheduled intervals, streaming systems
process each event or message as it occurs, enabling near-real-time insights
and actions.

Continuous processing::
  Data is processed immediately as it arrives. Streams have no defined end;
  they represent ongoing, unbounded sequences of events rather than finite
  data sets.

Low latency::
  Processing occurs with minimal delay, typically in milliseconds, making
  streaming suitable for time-sensitive workloads where batch windows are
  not acceptable.

Scalable throughput::
  Streaming systems distribute processing across multiple nodes to handle
  high-volume data flows, allowing throughput to scale horizontally as
  demand increases.

Common use cases for data streaming include near-real-time analytics such as
aggregating website clickstream data or sensor telemetry, event processing for
fraud detection and system monitoring, and data pipeline workloads such as
synchronizing databases or populating search indexes.

.E-commerce clickstream example
====
An e-commerce platform streams user clickstream data to analyse shopping
behaviour as it occurs, enabling personalised product recommendations and
dynamic pricing without waiting for nightly batch reports.
====

{reactive-messaging-spec-name} supports data streaming through its
integration with messaging systems such as Apache Kafka, providing a
declarative programming model for building streaming applications on top
of MicroProfile {mp-version} runtimes.

=== Using Event Sourcing for Storing Changes

Event sourcing is a pattern where all changes to application state are stored as a sequence of events, rather than storing only the current state. Instead of updating a record in place, the system appends events describing what happened.

*Core Concepts of Event Sourcing*:

* *Event Store*: A database or log that stores all events in the order they occurred. Common implementations include dedicated event stores (EventStoreDB, Apache Kafka) or relational databases with an events table.
* *Immutable Events*: Once written, events cannot be changed, providing an authoritative audit trail. Events are facts about what happened, not intentions or commands.
* *Event Replay*: The current state can be reconstructed by replaying events from the beginning or from the last snapshot.
* *Aggregate*: A domain object that processes commands and produces events. The aggregate enforces business rules before emitting events.

*Benefits of Event Sourcing*:

* *Complete Audit Trail*: Every state change is recorded with full context (timestamp, user, reason), enabling compliance with regulations like PCI DSS, SOX, and GDPR.
* *Temporal Queries*: You can query what the state was at any point in time by replaying events up to that moment.
* *Event Replay*: Bugs in business logic can be fixed and events replayed to correct state without data loss.
* *Multiple Projections*: Different views or read models can be created from the same event stream, supporting CQRS (Command Query Responsibility Segregation) patterns.
* *Natural Integration*: Events can be published to message brokers for reactive processing by other microservices.

*Challenges and Considerations*:

* *Complexity*: Event sourcing adds architectural complexity compared to traditional CRUD operations.
* *Event Schema Evolution*: As the system evolves, you need strategies to handle old event versions (upcasting, versioning).
* *Performance*: Replaying thousands of events to rebuild state can be slow. Snapshots (periodic state checkpoints) mitigate this.
* *Eventual Consistency*: Read models derived from events may lag behind writes, requiring careful UX design.

*Example*: In a banking application, instead of storing just the current account balance in a single row, the event-sourced system stores immutable events:

----
AccountOpened(accountId: "123", initialBalance: 1000, timestamp: "2024-01-01T10:00:00Z")
AccountCredited(accountId: "123", amount: 100, timestamp: "2024-01-02T14:30:00Z")
AccountDebited(accountId: "123", amount: 50, timestamp: "2024-01-03T09:15:00Z")
----

The current balance (1050) is computed by replaying all events. If you need the balance as of January 2nd, replay events up to that date (1100). The event store becomes the authoritative source of truth, while the current balance is a projection that can be rebuilt at any time.

*Integration with MicroProfile*: Event sourcing pairs naturally with MicroProfile capabilities for building reactive, auditable microservices:

* *MicroProfile Reactive Messaging*: When domain events are persisted to the event store, they can be simultaneously published to Kafka topics using `@Outgoing` channels. Other microservices subscribe to these events with `@Incoming` to maintain their own projections or trigger downstream workflows. For example, when an `AccountDebited` event is stored, it's published to a Kafka topic where a fraud detection service consumes it to check for suspicious patterns.

* *MicroProfile GraphQL*: Event streams can be exposed through GraphQL subscriptions, allowing clients to receive real-time updates as events occur. Temporal queries ("what was the account state on January 2nd?") can be implemented as GraphQL queries that replay events up to a specified timestamp.

* *MicroProfile Fault Tolerance*: Apply `@Retry` and `@CircuitBreaker` patterns when persisting events to ensure the event store remains consistent even during transient failures. Since events are immutable, idempotency strategies ensure duplicate event writes are safely handled.

* *MicroProfile Telemetry*: Instrument event processing with tracing spans to track event flow from command receipt through event persistence and projection updates. Emit metrics on event throughput, replay performance, and projection lag to monitor system health.

This combination enables microservices to maintain complete audit trails while staying responsive and resilient through asynchronous event processing.

=== Rebuilding State from Events

One of the powerful features of event sourcing is the ability to rebuild application state by replaying events. This process involves:

1. *Starting with an Initial State*: Begin with an empty or default state for the aggregate or projection.
2. *Applying Events Sequentially*: Process each event in order, updating the state according to the event's semantics.
3. *Reaching the Current State*: After processing all events, the system arrives at the current state.

*Benefits of State Rebuilding*:

* *Disaster Recovery*: If the current state is lost or corrupted, it can be reconstructed from events without data loss.
* *New Projections*: New read models or views can be created by replaying events with different logic, enabling new features without migrating data.
* *Debugging and Testing*: Historical states can be recreated to understand bugs or test scenarios. Developers can replay production events in development environments to reproduce issues.
* *Event Schema Migration*: When event formats change, replay events through migration logic to update projections to the new schema.

*Optimization with Snapshots*:

For systems with long event histories, replaying every event can be time-consuming. Snapshots periodically capture the current state at a specific event sequence number, allowing rebuilding to start from the snapshot and replay only subsequent events. This dramatically reduces recovery time for aggregates with thousands of events.

The snapshot strategy balances storage costs against replay performance. Common approaches include snapshotting every N events (for example, every 100 events) or after significant state changes (for example, order completion).

*Example*: An order management system stores events in sequence:

----
OrderCreated(orderId: "789", customerId: "C123", timestamp: "2024-01-15T09:00:00Z")
ItemAdded(orderId: "789", productId: "P456", quantity: 2, timestamp: "2024-01-15T09:05:00Z")
ItemAdded(orderId: "789", productId: "P789", quantity: 1, timestamp: "2024-01-15T09:07:00Z")
PaymentProcessed(orderId: "789", amount: 150.00, timestamp: "2024-01-15T10:30:00Z")
OrderShipped(orderId: "789", trackingNumber: "TRK123", timestamp: "2024-01-16T14:00:00Z")
----

To rebuild the current state of order "789", replay these events in sequence, updating the order's status and contents at each step. The final state shows a shipped order with two line items and confirmed payment.

*Integration with MicroProfile*:

* *MicroProfile Reactive Messaging for Event Replay*: Consume events from Kafka topics using `@Incoming` channels to build projections. When creating a new projection, use Kafka consumer group offsets to replay events from the beginning of the topic. For example, a reporting service can subscribe to order events and build a sales dashboard by processing the entire event history, then continue receiving live updates.

* *MicroProfile Config for Snapshot Strategy*: Configure snapshot intervals using MicroProfile Config properties such as `snapshot.interval.events=100` or `snapshot.enabled=true`. This allows operations teams to tune snapshot frequency based on system performance without code changes.

* *MicroProfile Metrics for Replay Monitoring*: Track replay progress with metrics like `events_replayed_total`, `replay_duration_seconds`, and `projection_lag_events` to monitor how far projections lag behind the event stream. These metrics help identify when snapshots are needed or when projection logic needs optimization.

* *MicroProfile Fault Tolerance during Replay*: Apply `@Retry` and `@Timeout` to event replay operations to handle transient failures when reading from event stores. Use `@Bulkhead` to limit concurrent replay operations and prevent resource exhaustion when rebuilding multiple projections simultaneously.

This combination of MicroProfile specifications enables robust, observable event sourcing implementations. Event stores like Apache Kafka provide durable, ordered event streams, while MicroProfile Reactive Messaging handles consumption and processing. MicroProfile Config, Metrics, and Fault Tolerance add production-ready operational capabilities around event replay and projection management.
