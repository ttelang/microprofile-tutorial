== Overview

This chapter demonstrates the implementation of *MicroProfile Metrics 5.1* in a Jakarta EE 10 application running on Open Liberty. The Catalog Service shows how to instrument microservices with comprehensive performance monitoring using both annotation-based and programmatic metrics.

MicroProfile Metrics provides a standardized way to expose application metrics in formats compatible with monitoring tools like Prometheus and Grafana. This is essential for:

* Performance monitoring and optimization
* Capacity planning and resource management
* Identifying bottlenecks and troubleshooting issues
* Real-time operational insights
* Integration with container orchestration platforms

== Prerequisites

Before working with this chapter, you should be familiar with:

* *Chapter 5* - MicroProfile Config for externalized configuration
* *Chapter 6* - MicroProfile Health for health checks
* Basic understanding of monitoring and observability concepts

This chapter focuses exclusively on implementing and configuring MicroProfile Metrics.

== MicroProfile Metrics Implementation

=== Metrics Types Implemented

This application demonstrates all four MicroProfile Metrics types:

1. *Counter* - Monotonically increasing values (request counts, items created)
2. *Gauge* - Values that go up and down (catalog size, active connections)
3. *Timer* - Duration and count of events (method execution time)
4. *Histogram* - Distribution of values (response payload sizes)

=== Annotation-Based Metrics

==== Counter Metric - Tracking Request Count

Track the number of times the product list is accessed:

[source,java]
----
@GET
@Produces(MediaType.APPLICATION_JSON)
@Counted(
    name = "productAccessCount",
    absolute = true,
    description = "Number of times the list of products is requested"
)
public Response getAllProducts() {
    List<Product> products = productService.findAllProducts();
    return Response.ok(products).build();
}
----

*Access the metric:*
[source,bash]
----
curl http://localhost:5050/metrics?scope=application
----

*Output:*
[source]
----
# HELP productAccessCount_total Number of times the list of products is requested
# TYPE productAccessCount_total counter
productAccessCount_total{mp_scope="application",} 5.0
----

==== Timer Metric - Tracking Response Time

Monitor how long it takes to look up products:

[source,java]
----
@GET
@Path("/{id}")
@Produces(MediaType.APPLICATION_JSON)
@Timed(
    name = "productLookupTime",
    tags = {"operation=read", "resource=product"},
    absolute = true, 
    description = "Time spent looking up products by ID"
)
public Response getProductById(@PathParam("id") Long id) {
    Product product = productService.findProductById(id);
    return product != null ? Response.ok(product).build() 
                          : Response.status(Response.Status.NOT_FOUND).build();
}
----

*Output:*
[source]
----
# HELP productLookupTime_seconds_max Time spent looking up products by ID
# TYPE productLookupTime_seconds_max gauge
productLookupTime_seconds_max{operation="read",resource="product",mp_scope="application",} 0.002270643
# HELP productLookupTime_seconds Time spent looking up products by ID
# TYPE productLookupTime_seconds summary
productLookupTime_seconds_count{operation="read",resource="product",mp_scope="application",} 10.0
productLookupTime_seconds_sum{operation="read",resource="product",mp_scope="application",} 0.015234
----

*Tag Usage:*
Tags add dimensions to metrics, allowing you to filter and aggregate by:
- `operation`: read, create, update, delete
- `resource`: product, order, user
- Custom business dimensions

==== Gauge Metric - Real-time Catalog Size

Monitor the current number of products in the catalog:

[source,java]
----
@GET
@Path("/count")
@Produces(MediaType.APPLICATION_JSON)
@Gauge(
    name = "productCatalogSize", 
    unit = "none", 
    description = "Current number of products in the catalog"
)
public long getProductCount() {
    return productService.findAllProducts().size();
}
----

The gauge is evaluated each time the `/metrics` endpoint is accessed, providing real-time values.

*Output:*
[source]
----
# HELP io_microprofile_tutorial_store_product_resource_ProductResource_productCatalogSize Current number of products in the catalog
# TYPE io_microprofile_tutorial_store_product_resource_ProductResource_productCatalogSize gauge
io_microprofile_tutorial_store_product_resource_ProductResource_productCatalogSize{mp_scope="application",} 8.0
----

IMPORTANT: Gauge methods are called on every metrics scrape. Avoid expensive operations in gauge methods. For costly calculations, cache the value and update it periodically.

=== Programmatic Metrics API

The programmatic API provides more flexibility and control over metrics registration and usage.

==== Injecting MetricRegistry

[source,java]
----
import org.eclipse.microprofile.metrics.MetricRegistry;
import org.eclipse.microprofile.metrics.annotation.RegistryScope;

@ApplicationScoped
public class ProductResource {
    
    @Inject
    @RegistryScope(scope = MetricRegistry.APPLICATION_SCOPE)
    private MetricRegistry registry;
    
    private Histogram responsePayloadHistogram;
    
    @PostConstruct
    public void initMetrics() {
        // Register metrics programmatically
        responsePayloadHistogram = registry.histogram(
            Metadata.builder()
                .withName("product.response.payload.size")
                .withDescription("Distribution of product list response sizes")
                .withUnit(MetricUnits.BYTES)
                .build()
        );
    }
}
----

==== Histogram - Tracking Response Size Distribution

Monitor the distribution of response payload sizes:

[source,java]
----
@GET
@Produces(MediaType.APPLICATION_JSON)
@Counted(name = "productAccessCount", absolute = true,
    description = "Number of times the list of products is requested")
public Response getAllProducts() {
    List<Product> products = productService.findAllProducts();
    
    // Track response payload size distribution
    String json = JsonbBuilder.create().toJson(products);
    responsePayloadHistogram.update(json.getBytes().length);
    
    return Response.ok(products).build();
}
----

Histograms track statistical distributions including:
- Count, min, max, mean
- Standard deviation
- Percentiles (p50, p75, p95, p98, p99, p999)

==== Service-Level Metrics

The `ProductService` uses programmatic metrics to track business operations:

[source,java]
----
@ApplicationScoped
public class ProductService {
    
    @Inject
    @RegistryScope(scope = MetricRegistry.APPLICATION_SCOPE)
    private MetricRegistry registry;
    
    private Counter productCreatedCounter;
    private Counter productDeletedCounter;
    private Timer searchTimer;
    
    @PostConstruct
    public void initMetrics() {
        // Register counters
        productCreatedCounter = registry.counter(
            Metadata.builder()
                .withName("products.created.total")
                .withDescription("Total number of products created")
                .build()
        );
        
        productDeletedCounter = registry.counter(
            Metadata.builder()
                .withName("products.deleted.total")
                .withDescription("Total number of products deleted")
                .build()
        );
        
        // Register timer
        searchTimer = registry.timer(
            Metadata.builder()
                .withName("product.search.duration")
                .withDescription("Time taken to execute product search")
                .withUnit(MetricUnits.MILLISECONDS)
                .build()
        );
    }
    
    public Product createProduct(Product product) {
        Product created = repository.createProduct(product);
        productCreatedCounter.inc(); // Track creation
        return created;
    }
    
    public boolean deleteProduct(Long id) {
        boolean deleted = repository.deleteProduct(id);
        if (deleted) {
            productDeletedCounter.inc(); // Track deletion
        }
        return deleted;
    }
    
    public List<Product> searchProducts(String name, String description, 
                                       Double minPrice, Double maxPrice) {
        // Measure search performance
        Timer.Context context = searchTimer.time();
        try {
            return repository.searchProducts(name, description, minPrice, maxPrice);
        } finally {
            context.stop();
        }
    }
}
----

=== Annotation-Based vs Programmatic Metrics

*Use Annotations when:*
- Metrics are static and known at compile time
- You want simpler, more readable code
- Metrics align with specific methods

*Use Programmatic API when:*
- Metrics need to be created dynamically at runtime
- You need more control over metric metadata
- Metrics don't align with specific method boundaries
- Working with legacy code or third-party libraries

== Accessing Metrics

=== Metrics Endpoints

MicroProfile Metrics exposes several endpoints:

[source,bash]
----
# All metrics from all scopes
GET /metrics

# Application-specific metrics
GET /metrics?scope=application

# Base metrics (JVM, runtime)
GET /metrics?scope=base

# Vendor-specific metrics
GET /metrics?scope=vendor

# Specific metric by name
GET /metrics?scope=application&name=productAccessCount
----

=== Output Format

MicroProfile Metrics 5.x uses Prometheus format as the standard output format. JSON format is optional in the specification and is not supported by Open Liberty's implementation.

*Accessing metrics in Prometheus format:*

[source,bash]
----
curl http://localhost:5050/metrics?scope=application
----

Output:
[source]
----
# TYPE productAccessCount_total counter
# HELP productAccessCount_total Number of times the list of products is requested
productAccessCount_total{mp_scope="application",} 5.0

# TYPE products_created_total counter
# HELP products_created_total Total number of products created
products_created_total{mp_scope="application",} 3.0

# TYPE product_response_payload_size_bytes summary
# HELP product_response_payload_size_bytes Distribution of product list response sizes
product_response_payload_size_bytes_count{mp_scope="application",} 5.0
product_response_payload_size_bytes_sum{mp_scope="application",} 12456.0
product_response_payload_size_bytes{mp_scope="application",quantile="0.5",} 2489.0
product_response_payload_size_bytes{mp_scope="application",quantile="0.75",} 2501.0
product_response_payload_size_bytes{mp_scope="application",quantile="0.95",} 2512.0
----

NOTE: In MicroProfile Metrics 5.x, JSON format is optional and not supported by all implementations. Open Liberty uses Prometheus format exclusively. For programmatic access, parse the Prometheus text format or use monitoring tools like Prometheus and Grafana.

== Metric Scopes

MicroProfile Metrics organizes metrics into three scopes:

=== Application Scope

Custom metrics defined by the application developer:

[source,bash]
----
curl http://localhost:5050/metrics?scope=application
----

Contains:
- `productAccessCount`
- `products.created.total`
- `products.deleted.total`
- `productLookupTime`
- `product.response.payload.size`
- `product.search.duration`
- `productCatalogSize`

=== Base Scope (Optional in 5.x)

Standard JVM and runtime metrics:

[source,bash]
----
curl http://localhost:5050/metrics?scope=base
----

Contains:
- Memory usage (heap, non-heap)
- Thread counts
- Class loading statistics
- JVM uptime
- CPU usage

=== Vendor Scope

Implementation-specific metrics from Open Liberty:

[source,bash]
----
curl http://localhost:5050/metrics?scope=vendor
----

Contains Open Liberty runtime metrics like connection pools, servlet statistics, etc.

== Configuration

=== Server Configuration (server.xml)

Enable MicroProfile Metrics in Open Liberty:

[source,xml]
----
<featureManager>
    <platform>microProfile-7.1</platform>
    <feature>mpMetrics</feature>
</featureManager>

<!-- Disable authentication for development -->
<mpMetrics authentication="false" />
----

IMPORTANT: In production, enable authentication and restrict access to metrics endpoints.

=== Maven Dependencies

The MicroProfile 7.1 platform includes metrics:

[source,xml]
----
<dependency>
    <groupId>org.eclipse.microprofile</groupId>
    <artifactId>microprofile</artifactId>
    <version>7.1</version>
    <type>pom</type>
    <scope>provided</scope>
</dependency>
----

For standalone usage:

[source,xml]
----
<dependency>
    <groupId>org.eclipse.microprofile.metrics</groupId>
    <artifactId>microprofile-metrics-api</artifactId>
    <version>5.1.1</version>
    <scope>provided</scope>
</dependency>
----

[source,yaml]
----
scrape_configs:
  - job_name: 'catalog-service'
    metrics_path: '/metrics'
    scrape_interval: 15s
    static_configs:
      - targets: ['localhost:5050']

=== Grafana Dashboard

Create dashboards using Prometheus as a data source:

1. Add Prometheus data source in Grafana
2. Create dashboard panels with PromQL queries:

[source,promql]
----
# Request rate
rate(productAccessCount_total[5m])

# Average response time
rate(productLookupTime_seconds_sum[5m]) / rate(productLookupTime_seconds_count[5m])

# 95th percentile response time
histogram_quantile(0.95, rate(productLookupTime_seconds_bucket[5m]))

productCatalogSize
----

=== Kubernetes Integration

Use metrics for horizontal pod autoscaling:

[source,yaml]
----
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: catalog-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: catalog-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metric:
        name: productAccessCount_total
      target:
        type: AverageValue
        averageValue: "1000"
----

== Best Practices

=== Metric Naming Conventions

Follow these conventions for consistent, discoverable metrics:

* Use lowercase with dots: `product.access.count`
* Start with domain/component: `catalog.product.search.time`
* Add suffixes for clarity:
  - `.total` for counters: `requests.total`
  - `.current` for gauges: `connections.current`
  - `.duration` for timers: `processing.duration`
  - `.size` for histograms: `payload.size`

=== Performance Considerations

* *Keep metrics lightweight*: Avoid expensive operations in gauge methods
* *Use appropriate types*: Don't use Timer when Counter suffices
* *Limit cardinality*: Avoid high-cardinality tags (user IDs, timestamps)
* *Cache gauge values*: Update periodically rather than compute on every scrape

=== When to Use Each Metric Type

[cols="1,2,2", options="header"]
|===
| Metric Type | Use Cases | Examples

| *Counter*
| Monotonically increasing values
| Requests processed, items created, errors occurred

| *Gauge*
| Values that fluctuate
| Queue size, active sessions, memory usage, catalog size

| *Timer*
| Duration and count with statistics
| Method execution time, request processing, database queries

| *Histogram*
| Value distributions
| Response sizes, batch sizes, payload distributions
|===

=== Tag Usage

Use tags to add dimensions without creating separate metrics:

[source,java]
----
// Good - Single metric with tags
@Timed(
    name = "product.operations.duration",
    tags = {"operation=read", "resource=product"}
)

// Bad - Separate metrics for each operation
@Timed(name = "product.read.duration")
@Timed(name = "product.create.duration")
----

=== Security Considerations

* Enable authentication for metrics endpoints in production
* Use network policies to restrict access
* Don't include sensitive data in metric names or tags
* Expose metrics only to internal monitoring systems
