= Payment Service

This microservice is part of the Jakarta EE 10 and MicroProfile-based e-commerce application. It handles payment processing, idempotency, audit logging, and demonstrates **advanced MicroProfile Config patterns**.

== Overview

The Payment Service showcases advanced MicroProfile Config capabilities:

* **@ConfigProperties** - Grouping related configuration properties into a configuration bean
* **Custom ConfigSource** - Creating custom configuration sources with specific ordinality
* **External Integration Configuration** - Managing complex third-party service settings
* **Security-Sensitive Configuration** - Handling API keys and credentials
* **Payment Processing** - Transaction processing with idempotency support
* **Audit Trail** - Comprehensive audit logging for compliance and troubleshooting
* **CDI Interceptors** - Automatic logging using interceptor patterns
* **RESTful API** - Complete OpenAPI-documented endpoints
* **Gateway Integration** - External payment gateway connectivity

== Configuration Architecture

=== Grouped Configuration with @ConfigProperties

The Payment Service uses the **@ConfigProperties** annotation to group related payment gateway configuration into a single configuration bean. This is more maintainable than injecting individual properties.

==== PaymentGatewayConfig Bean

[source,java]
----
@ConfigProperties(prefix = "payment.gateway")
@ApplicationScoped
public class PaymentGatewayConfig {
    
    // Maps to: payment.gateway.endpoint
    public String endpoint;
    
    // Maps to: payment.gateway.apiKey
    public String apiKey;
    
    // Maps to: payment.gateway.timeout
    public int timeout;
    
    // Maps to: payment.gateway.retryAttempts
    public int retryAttempts;
    
    // Maps to: payment.gateway.sandboxMode
    public boolean sandboxMode;
}
----

==== Configuration Properties

Defined in `src/main/resources/META-INF/microprofile-config.properties`:

[source,properties]
----
# MicroProfile Config Properties for Payment Service

# OpenAPI Configuration
mp.openapi.scan.disabled=false

# Payment Gateway Configuration
# These properties are automatically mapped to PaymentGatewayConfig bean
payment.gateway.endpoint=https://api.paymentgateway.com/v1
payment.gateway.apiKey=demo-api-key-12345
payment.gateway.timeout=5000
payment.gateway.retryAttempts=3
payment.gateway.sandboxMode=true
----

==== Using Grouped Configuration

Instead of injecting individual properties:

[source,java]
----
// Individual property injection approach:
@Inject @ConfigProperty(name="payment.gateway.endpoint") String endpoint;
@Inject @ConfigProperty(name="payment.gateway.timeout") int timeout;
// ... more individual properties
----

The service injects the entire configuration group:

[source,java]
----
// Grouped configuration approach:
@ApplicationScoped
public class PaymentService {
    
    @Inject
    private PaymentGatewayConfig gatewayConfig;
    
    public boolean processPayment(PaymentDetails paymentDetails) {
        // Access configuration via the bean
        System.out.println("Using gateway: " + gatewayConfig.endpoint);
        System.out.println("Timeout: " + gatewayConfig.timeout + "ms");
        System.out.println("Sandbox mode: " + gatewayConfig.sandboxMode);
        // ...
    }
}
----

**Benefits of @ConfigProperties:**

* Cleaner code with less injection boilerplate
* Logical grouping of related configuration
* Single point of configuration for a feature
* Better testability and mockability
* Easier refactoring and maintenance

=== Custom ConfigSource

The Payment Service implements a **custom ConfigSource** named `PaymentServiceConfigSource` that provides payment-specific configuration with high priority (ordinal: 600).

==== Implementation

[source,java]
----
public class PaymentServiceConfigSource implements ConfigSource {
    
    private Map<String, String> properties = new HashMap<>();

    public PaymentServiceConfigSource() {
        // Load payment service configurations dynamically
        properties.put("payment.gateway.apiKey", "secret_api_key");
        properties.put("payment.gateway.endpoint", "https://api.paymentgateway.com");
    }

    @Override
    public Map<String, String> getProperties() {
        return properties;
    }

    @Override
    public String getValue(String propertyName) {
        return properties.get(propertyName);
    }

    @Override
    public String getName() {
        return "PaymentServiceConfigSource";
    }

    @Override
    public int getOrdinal() {
        // High priority to override default configurations
        return 600;
    }

    @Override
    public Set<String> getPropertyNames() {
        return properties.keySet();
    }
}
----

==== Registration

The custom ConfigSource is registered via the ServiceLoader mechanism in:
`src/main/resources/META-INF/services/org.eclipse.microprofile.config.spi.ConfigSource`

[source]
----
io.microprofile.tutorial.store.payment.config.PaymentServiceConfigSource
----

==== Configuration Source Priority

MicroProfile Config uses a prioritized set of configuration sources. The payment service configuration sources in order of priority (highest to lowest):

1. **Custom ConfigSource** (`PaymentServiceConfigSource`) - Ordinal: 600
2. System properties - Ordinal: 400
3. Environment variables - Ordinal: 300
4. microprofile-config.properties file - Ordinal: 100

**Important:** The custom ConfigSource provides `payment.gateway.endpoint` with ordinal 600, which means it will **override** system properties (400) and environment variables (300) for that specific property. This demonstrates how custom ConfigSources can enforce certain configurations that cannot be accidentally overridden by typical deployment-time settings.

To override a property from the custom ConfigSource, you would need to either:
- Create another ConfigSource with ordinal > 600
- Modify the custom ConfigSource itself
- Use properties that are NOT provided by the custom ConfigSource (e.g., timeout, retryAttempts, sandboxMode)

**Test 3: Override with System Property**

System properties (ordinal 400) override environment variables (ordinal 300).

To pass system properties to the Liberty server, you need to configure them in the `pom.xml` or use the `jvmOptions` parameter.

**Option 1: Using pom.xml configuration (Permanent)**

Edit `pom.xml` and add JVM options to the liberty-maven-plugin:

[source,xml]
----
<plugin>
    <groupId>io.openliberty.tools</groupId>
    <artifactId>liberty-maven-plugin</artifactId>
    <version>3.11.2</version>
    <configuration>
        <serverName>mpServer</serverName>
        <jvmOptions>
            <param>-Dpayment.gateway.timeout=7777</param>
        </jvmOptions>
    </configuration>
</plugin>
----

Then restart the server:

[source,bash]
----
# Step 1: Stop the server (press Ctrl+C)

# Step 2: Clear the environment variable from Test 2 (optional)
unset PAYMENT_GATEWAY_TIMEOUT

# Step 3: Start server with updated pom.xml
mvn liberty:dev
----

**Option 2: Using environment variable for JVM options (Temporary)**

[source,bash]
----
# Step 1: Stop the server (press Ctrl+C)

# Step 2: Set MAVEN_OPTS or Liberty-specific JVM options
export MAVEN_OPTS="-Dpayment.gateway.timeout=7777"

# Step 3: Start server
mvn liberty:dev
----

**Option 3: Using server.xml configuration**

Edit `src/main/liberty/config/server.xml` and add the `jvmOptions` element:

[source,xml]
----
<server description="MicroProfile Payment Service">
    <featureManager>
        <!-- existing features -->
    </featureManager>
    
    <!-- JVM Options for system properties -->
    <jvmOptions>-Dpayment.gateway.timeout=7777</jvmOptions>
    
    <!-- rest of configuration -->
</server>
----

You can also set multiple JVM options:

**Configuration Priority Summary:**

[cols="1,1,1,2", options="header"]
|===
| Priority | Source | Ordinal | Can Override
| 1 (Highest) | Custom ConfigSource | 600 | All sources below
| 2 | System Properties (-D flags) | 400 | Env vars, config file
| 3 | Environment Variables | 300 | Config file only
| 4 (Lowest) | microprofile-config.properties | 100 | None
|===

== Troubleshooting

=== Common Issues

==== Port Conflicts

If you encounter a port conflict when starting the server, you can change the ports in the `pom.xml` file:

[source,xml]
----
<liberty.var.default.http.port>9080</liberty.var.default.http.port>
<liberty.var.default.https.port>9081</liberty.var.default.https.port>
----

==== ConfigSource Not Loading

If the custom ConfigSource is not loading, check the following:

1. Verify the service provider configuration file exists at:
   `src/main/resources/META-INF/services/org.eclipse.microprofile.config.spi.ConfigSource`

2. Ensure it contains the correct fully qualified class name:
   `io.microprofile.tutorial.store.payment.config.PaymentServiceConfigSource`

==== Deployment Errors

For CWWKZ0004E deployment errors, check the server logs at:
`target/liberty/wlp/usr/servers/mpServer/logs/messages.log`

==== Payment Application Not Accessible (Context Root Not Found)

If you see "Context Root Not Found" or the OpenLiberty default page when accessing endpoints:

**Problem:** The payment application is not deployed or Liberty is running without the payment app.

**Solutions:**

1. **Verify the payment service is running:**
+
[source,bash]
----
# Make sure you're in the payment directory
cd /workspaces/microprofile-tutorial/code/chapter05/payment

# Check if liberty:dev is running
# You should see an active terminal with Liberty server output
----

2. **Check if the application is deployed:**
+
Look for this message in the Liberty server output:
+
[source,text]
----
[INFO] [AUDIT] CWWKZ0001I: Application payment started
----

3. **If the application is not started:**
+
[source,bash]
----
# Stop any running server (Ctrl+C)
# Clean and restart
mvn clean liberty:dev
----

4. **Verify the correct port:**
+
The payment service runs on port **9080**, not 5050 (which is used by the catalog service)

5. **Test with the root context:**
+
[source,bash]
----
# This should show the welcome page, not an error
curl http://localhost:9080/payment/

# This should return JSON config
curl http://localhost:9080/payment/api/payment-config
----

6. **Check for port conflicts:**
+
If port 9080 is in use, you'll see an error. Kill any other processes using that port or change the port in `pom.xml`
