== Introduction

In financial and e-commerce applications, maintaining a complete audit trail of all payment transactions isn't just a best practice—it's often a **regulatory requirement**. Whether you're dealing with PCI DSS compliance, SOX regulations, or GDPR data protection requirements, having detailed, immutable records of every payment operation is crucial for:

* **Compliance & Auditing** - Meeting regulatory requirements and passing audits
* **Fraud Detection** - Identifying suspicious patterns and unauthorized transactions
* **Dispute Resolution** - Providing evidence for chargebacks and customer disputes
* **Troubleshooting** - Debugging production issues and investigating failures
* **Business Analytics** - Understanding transaction patterns and success rates

In this blog post, we'll explore how to implement a robust audit logging system for a payment microservice using **Jakarta EE 10** and **MicroProfile 7.1**. Our implementation demonstrates practical patterns that can be applied to any enterprise application requiring comprehensive audit trails.

== The Challenge: What Should We Audit?

Before diving into implementation, let's identify what information we need to capture for each payment operation:

=== Essential Audit Data

[cols="1,2,1"]
|===
|Data Point |Purpose |Security Consideration

|Audit ID
|Unique identifier for each audit record
|None

|Payment ID
|Links audit records to specific transactions
|None

|Timestamp
|When the operation occurred
|Use server time (UTC)

|Operation Type
|What action was performed (process, validate, refund)
|None

|Status
|Success, failed, validation error, etc.
|None

|Card Number
|Payment method used
|**MUST be masked** (show last 4 digits only)

|Cardholder Name
|Who initiated the payment
|May need tokenization

|Amount
|Transaction value
|None

|Duration
|How long the operation took
|Performance metric

|Message
|Detailed description of outcome
|Sanitize error messages

|Cached Response
|Whether this was an idempotent replay
|Helps detect retry patterns

|Client Identifier
|Source of the request (IP, user agent)
|Consider privacy laws
|===

[IMPORTANT]
====
**Security First**: Never log sensitive data like full card numbers, CVV codes, or PINs in plain text. Our implementation uses masking to show only the last 4 digits of card numbers.
====

== Implementation Deep Dive

=== Step 1: Define the Audit Record Entity

First, we create a comprehensive data model to capture all relevant audit information:

[source,java]
----
package io.microprofile.tutorial.store.payment.entity;

import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;
import java.math.BigDecimal;
import java.time.LocalDateTime;

@Data
@AllArgsConstructor
@NoArgsConstructor
public class PaymentAuditRecord {
    
    private String auditId;              // Unique identifier
    private String paymentId;            // Transaction reference
    private AuditOperationType operationType;
    private LocalDateTime timestamp;     // When it happened
    private AuditStatus status;          // Outcome
    private String maskedCardNumber;     // Last 4 digits only!
    private String cardHolderName;
    private BigDecimal amount;
    private String message;              // Detailed description
    private String initiatedBy;          // User/system identifier
    private String clientIdentifier;     // IP address or client ID
    private Long durationMs;             // Performance tracking
    private Boolean cachedResponse;      // Idempotency flag
    private String metadata;             // Additional context
    
    // Operation types supported
    public enum AuditOperationType {
        PAYMENT_PROCESS,
        PAYMENT_VALIDATE,
        PAYMENT_REFUND,
        IDEMPOTENT_PAYMENT,
        IDEMPOTENT_CACHE_HIT
    }
    
    // Possible statuses
    public enum AuditStatus {
        SUCCESS,
        FAILED,
        VALIDATION_ERROR,
        CONFLICT,
        SYSTEM_ERROR
    }
    
    // Security: Mask card numbers
    public static String maskCardNumber(String cardNumber) {
        if (cardNumber == null || cardNumber.length() < 4) {
            return "****";
        }
        String lastFour = cardNumber.substring(cardNumber.length() - 4);
        return "**** **** **** " + lastFour;
    }
}
----

**Key Design Decisions:**

* **Enums for Operation Type and Status**: Provides type safety and makes queries easier
* **LocalDateTime**: Uses Java 8 time API for better timezone handling
* **Lombok Annotations**: Reduces boilerplate with `@Data`, `@AllArgsConstructor`, `@NoArgsConstructor`
* **Card Masking Method**: Static utility to mask sensitive card numbers

=== Step 2: Create the Audit Service

The audit service is the heart of our logging system:

[source,java]
----
package io.microprofile.tutorial.store.payment.service;

import jakarta.enterprise.context.ApplicationScoped;
import java.util.concurrent.ConcurrentHashMap;
import java.util.logging.Logger;

@ApplicationScoped
public class PaymentAuditService {
    
    private static final Logger AUDIT_LOGGER = Logger.getLogger("PaymentAudit");
    
    // In-memory storage (use database in production!)
    private final ConcurrentHashMap<String, PaymentAuditRecord> auditRecords = 
        new ConcurrentHashMap<>();
    
    /**
     * Log a payment operation to the audit trail
     */
    public PaymentAuditRecord logAudit(
            String paymentId,
            AuditOperationType operationType,
            AuditStatus status,
            PaymentDetails paymentDetails,
            String message,
            Long durationMs,
            Boolean cachedResponse) {
        
        String auditId = UUID.randomUUID().toString();
        LocalDateTime timestamp = LocalDateTime.now();
        
        // SECURITY: Mask sensitive card information
        String maskedCard = paymentDetails != null && 
                           paymentDetails.getCardNumber() != null
                ? PaymentAuditRecord.maskCardNumber(paymentDetails.getCardNumber())
                : "N/A";
        
        // Build the audit record
        PaymentAuditRecord auditRecord = new PaymentAuditRecord();
        auditRecord.setAuditId(auditId);
        auditRecord.setPaymentId(paymentId);
        auditRecord.setOperationType(operationType);
        auditRecord.setTimestamp(timestamp);
        auditRecord.setStatus(status);
        auditRecord.setMaskedCardNumber(maskedCard);
        auditRecord.setCardHolderName(
            paymentDetails != null ? paymentDetails.getCardHolderName() : "N/A"
        );
        auditRecord.setAmount(
            paymentDetails != null ? paymentDetails.getAmount() : BigDecimal.ZERO
        );
        auditRecord.setMessage(message);
        auditRecord.setDurationMs(durationMs);
        auditRecord.setCachedResponse(cachedResponse != null ? cachedResponse : false);
        
        // Store the record
        auditRecords.put(auditId, auditRecord);
        
        // Also log to standard logger for immediate visibility
        AUDIT_LOGGER.log(Level.INFO, 
                "AUDIT: [{0}] {1} - PaymentId: {2}, Status: {3}, Card: {4}, " +
                "Amount: {5}, Duration: {6}ms, Message: {7}",
                new Object[]{
                    timestamp, operationType, paymentId, status,
                    maskedCard, amount, durationMs, message
                });
        
        return auditRecord;
    }
    
    // Query methods...
    public List<PaymentAuditRecord> getAllAuditRecords() { /*...*/ }
    public List<PaymentAuditRecord> getAuditRecordsByPaymentId(String paymentId) { /*...*/ }
    public List<PaymentAuditRecord> getAuditRecordsByOperationType(AuditOperationType type) { /*...*/ }
    public List<PaymentAuditRecord> getAuditRecordsByStatus(AuditStatus status) { /*...*/ }
}
----

**Key Features:**

* **`@ApplicationScoped`**: Single instance shared across the application
* **ConcurrentHashMap**: Thread-safe in-memory storage (use database for production)
* **Security-First**: Always masks card numbers before storing
* **Dual Logging**: Writes to both audit storage and standard logger
* **Rich Query API**: Multiple methods to retrieve audit records

[NOTE]
====
In production, replace the `ConcurrentHashMap` with a persistent database (PostgreSQL, MongoDB, etc.) or a dedicated audit service. Consider using write-ahead logging or event sourcing for critical financial data.
====

=== Step 3: Integrate Audit Logging into Business Logic

Now we inject the audit service into our payment service and call it at critical points:

[source,java]
----
@ApplicationScoped
@Logged  // CDI interceptor for method logging
public class PaymentService {
    
    @Inject
    private PaymentAuditService auditService;
    
    @Inject
    private IdempotencyService idempotencyService;
    
    /**
     * Process a payment with full audit trail
     */
    public boolean processPayment(PaymentDetails paymentDetails) {
        long startTime = System.currentTimeMillis();
        
        // Validate payment details
        if (!validatePaymentDetails(paymentDetails)) {
            long duration = System.currentTimeMillis() - startTime;
            
            // Audit the validation failure
            auditService.logAudit(
                paymentDetails.getPaymentId(),
                AuditOperationType.PAYMENT_PROCESS,
                AuditStatus.VALIDATION_ERROR,
                paymentDetails,
                "Payment validation failed: invalid card details",
                duration,  
                false
            );
            return false;
        }
        
        // Process the payment (simulate)
        boolean success = true; // In real world, call payment gateway
        long duration = System.currentTimeMillis() - startTime;
        
        // Audit the processing result
        auditService.logAudit(
            paymentDetails.getPaymentId(),
            AuditOperationType.PAYMENT_PROCESS,
            success ? AuditStatus.SUCCESS : AuditStatus.FAILED,
            paymentDetails,
            success ? "Payment processed successfully" : "Payment processing failed",
            duration,
            false
        );
        
        return success;
    }
    
    /**
     * Process idempotent payment with audit trail
     */
    public IdempotencyRecord processPaymentIdempotent(
            String paymentId, PaymentDetails paymentDetails) {
        long startTime = System.currentTimeMillis();
        
        // Check if already processed
        var existingRecord = idempotencyService.getExistingRecord(paymentId);
        
        if (existingRecord.isPresent()) {
            long duration = System.currentTimeMillis() - startTime;
            
            // Audit the cache hit
            auditService.logAudit(
                paymentId,
                AuditOperationType.IDEMPOTENT_CACHE_HIT,
                existingRecord.get().isSuccess() ? AuditStatus.SUCCESS : AuditStatus.FAILED,
                paymentDetails,
                "Idempotent request - returning cached result",
                duration,
                true  // This is a cached response
            );
            
            return existingRecord.get();
        }
        
        // Process new payment...
        boolean success = processPayment(paymentDetails);
        long duration = System.currentTimeMillis() - startTime;
        
        // Create and store idempotency record
        IdempotencyRecord record = idempotencyService.createRecord(
            paymentId, paymentDetails, success
        );
        
        // Audit the new idempotent payment
        auditService.logAudit(
            paymentId,
            AuditOperationType.IDEMPOTENT_PAYMENT,
            success ? AuditStatus.SUCCESS : AuditStatus.FAILED,
            paymentDetails,
            "Idempotent payment processed",
            duration,
            false
        );
        
        return record;
    }
}
----

**Audit Integration Points:**

. **Validation Failures**: Log immediately when validation fails
. **Processing Results**: Log both success and failure outcomes
. **Idempotency Cache Hits**: Track when cached responses are returned
. **Performance Metrics**: Capture operation duration for every call

=== Step 4: Expose Audit Data via REST API

Create a dedicated REST resource for querying audit trails:

[source,java]
----
@ApplicationScoped
@Path("/audit")
public class PaymentAuditResource {
    
    @Inject
    private PaymentAuditService auditService;
    
    /**
     * Get all audit records
     */
    @GET
    @Path("/records")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getAllAuditRecords() {
        List<PaymentAuditRecord> records = auditService.getAllAuditRecords();
        
        Map<String, Object> response = new HashMap<>();
        response.put("totalRecords", records.size());
        response.put("records", records);
        
        return Response.ok(response).build();
    }
    
    /**
     * Get audit records for a specific payment
     */
    @GET
    @Path("/payment/{paymentId}")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getAuditRecordsByPaymentId(@PathParam("paymentId") String paymentId) {
        List<PaymentAuditRecord> records = 
            auditService.getAuditRecordsByPaymentId(paymentId);
        
        if (records.isEmpty()) {
            return Response.status(Response.Status.NOT_FOUND)
                .entity(Map.of(
                    "message", "No audit records found",
                    "paymentId", paymentId
                ))
                .build();
        }
        
        return Response.ok(Map.of(
            "paymentId", paymentId,
            "totalRecords", records.size(),
            "records", records
        )).build();
    }
    
    /**
     * Get audit records by operation type
     */
    @GET
    @Path("/operation/{operationType}")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getAuditRecordsByOperationType(
            @PathParam("operationType") String operationType) {
        
        AuditOperationType type = AuditOperationType.valueOf(operationType.toUpperCase());
        List<PaymentAuditRecord> records = 
            auditService.getAuditRecordsByOperationType(type);
        
        return Response.ok(Map.of(
            "operationType", type,
            "totalRecords", records.size(),
            "records", records
        )).build();
    }
    
    /**
     * Get audit records by status
     */
    @GET
    @Path("/status/{status}")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getAuditRecordsByStatus(@PathParam("status") String status) {
        AuditStatus auditStatus = AuditStatus.valueOf(status.toUpperCase());
        List<PaymentAuditRecord> records = 
            auditService.getAuditRecordsByStatus(auditStatus);
        
        return Response.ok(Map.of(
            "status", auditStatus,
            "totalRecords", records.size(),
            "records", records
        )).build();
    }
}

== Production Considerations

=== 1. Persistent Storage

Replace in-memory storage with a database:

[source,java]
----
@ApplicationScoped
public class PaymentAuditService {
    
    @PersistenceContext
    private EntityManager em;
    
    @Transactional
    public PaymentAuditRecord logAudit(...) {
        // Create audit record
        PaymentAuditRecord record = new PaymentAuditRecord();
        // ... set properties ...
        
        // Persist to database
        em.persist(record);
        
        return record;
    }
}
----

**Recommended Databases:**

* **PostgreSQL**: Excellent for relational audit data with JSONB for metadata
* **MongoDB**: Good for high-volume writes and flexible schema
* **Elasticsearch**: Ideal for searching and analyzing large audit datasets
* **Dedicated Audit Services**: Consider AWS CloudTrail, Azure Monitor, or Splunk

=== 2. Asynchronous Logging

For high-throughput systems, make audit logging non-blocking:

[source,java]
----
@ApplicationScoped
public class PaymentAuditService {
    
    @Inject
    @Channel("audit-events")
    Emitter<PaymentAuditRecord> auditEmitter;
    
    public void logAudit(...) {
        PaymentAuditRecord record = createAuditRecord(...);
        
        // Send to async channel (MicroProfile Reactive Messaging)
        auditEmitter.send(record);
    }
}
----

=== 3. Retention Policies

Implement data retention based on compliance requirements:

=== 5. Immutability

Ensure audit records cannot be modified:

* Use append-only storage
* Implement write-once patterns
* Add cryptographic signatures
* Use blockchain for critical transactions

=== 6. Performance Optimization

For high-volume systems:

* **Batch Writes**: Group multiple audit records
* **Connection Pooling**: Optimize database connections
* **Indexing**: Index commonly queried fields (paymentId, timestamp, status)
* **Partitioning**: Partition by date for faster queries
* **Caching**: Cache frequent queries (but not the audit writes!)

== Conclusion

Building a comprehensive audit trail for payment services is not just about compliance—it's about building trust, enabling transparency, and providing the tools needed to run a reliable, secure payment platform.

Our implementation demonstrates several enterprise patterns:

✅ **Security-First Design**: Sensitive data is always masked +
✅ **Thread-Safe Operations**: Safe for concurrent access +
✅ **Rich Query API**: Multiple ways to retrieve audit data +
✅ **Performance Tracking**: Built-in operation timing +
✅ **Idempotency Awareness**: Tracks cached vs. fresh responses +
✅ **RESTful Access**: Easy integration with monitoring tools

=== Key Takeaways

. **Never log sensitive data in plain text** - Always mask card numbers, CVVs, etc.
. **Log early and often** - Capture failures, not just successes
. **Make audit data queryable** - Provide REST APIs for analysis
. **Plan for scale** - Use async logging and persistent storage for production
. **Think long-term** - Implement retention policies from day one
. **Secure audit access** - Apply strict role-based access control
