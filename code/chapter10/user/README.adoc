This service demonstrates JWT-secured user profile access in the MicroProfile tutorial.

== What The Service Does

The current implementation exposes one secured endpoint that reads claims from the JWT token and returns a simple profile string.

* Context root: `/user`
* JAX-RS base path: `/api`
* Resource path: `/users`
* Endpoint path: `/user-profile`

Full endpoint URL:

[source,text]
----
GET http://localhost:6050/user/api/users/user-profile
----

== Technology Used

* MicroProfile 7.1
* Jakarta RESTful Web Services
* MicroProfile JWT
* MicroProfile OpenAPI
* Maven

== Run The Service Locally

From this directory (`code/chapter10/user`):

[source,bash]
----
mvn clean package
mvn liberty:run
----

The service listens on port `6050`.

* Service landing page: `http://localhost:6050/user/`

== JWT Requirements

The endpoint requires a valid Bearer token and role `user`.

Required and expected claims:

* `iss`: must be `mp-ecomm-store`
* `sub`: user id (for example `user1`)
* `groups`: must include `user`
* `tenant_id`: tenant value shown in response
* `upn`: principal displayed in response (for example `user1@example.com`)

Header format:

[source,text]
----
Authorization: Bearer <jwt-token>
----

== Manual Testing (No Script)

Run these commands and verify the output.

=== 1. Start the service

This will run the service endpoint so that you can validate authentication.

[source,bash]
----
cd code/chapter10/user
mvn liberty:run
----

=== 2. Generate a JWT token manually

This step generates a JWT token required for accessing the secured endpoint.

[source,bash]
----
cd ../tools
java -jar jwtenizr.jar
----

This command writes a signed token to `code/chapter10/tools/token.jwt`.

=== 3. Inspect the token payload template (optional but recommended)

This confirms the claims match what the service expects (`iss`, `groups`, `upn`, `tenant_id`).

[source,bash]
----
cat jwt-token.json
----

=== 4. Call the secured endpoint with curl

This step verifies end-to-end JWT authentication and claim extraction.

[source,bash]
----
curl -H "Authorization: Bearer $(cat token.jwt)" \
  http://localhost:6050/user/api/users/user-profile
----

Expected response:

[source,text]
----
User: user1@example.com, Roles: [user], Tenant: ecomm-tenant-1
----

=== 5. Negative checks (manual verification)

This step helps confirm security behavior is correct.

No token (expect 401):

[source,bash]
----
curl -i http://localhost:6050/user/api/users/user-profile
----

Wrong role (expect 403):

1. Edit `jwt-token.json` and replace `"user"` with a different role.
2. Run `java -jar jwtenizr.jar` again to regenerate `token.jwt`.
3. Retry the same curl command.

== Manual Testing In Swagger UI (No Script)

1. Open `http://localhost:6050/openapi/ui`.
2. Click `Authorize`.
3. Paste the token from `code/chapter10/tools/token.jwt` (with or without `Bearer ` prefix).
4. Execute `GET /user/api/users/user-profile`.

This step confirms the OpenAPI security scheme and lets you test the same secured endpoint from a browser.

== Common Issues

* `401 Unauthorized`: token missing, expired, or invalid signature.
* `403 Forbidden`: token is valid, but `groups` does not contain `user`.
* Wrong URL: use `/user/api/users/user-profile` (not `/user/users/user-profile`).
