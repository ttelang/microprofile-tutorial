= Store Order Service with Reactive Messaging

This example demonstrates MicroProfile Reactive Messaging 3.0.1.

== Overview

The existing order creation path is used as the entry point, and Reactive Messaging is introduced so payment and other downstream processing can happen asynchronously.

This project showcases:

* **Message transformation** with `@Incoming` and `@Outgoing` annotations
* **Programmatic message emission** using `@Emitter` and `@Channel`
* **Message acknowledgment strategies** with manual control
* **Async payment hand-off** using `CompletionStage`
* **Stream filtering** for high-value paid orders

For the full payment-side implementation, see `payment/` service example.

== Code Examples

=== 1. PriceConverter - Basic Message Transformation

Demonstrates the simplest form of reactive messaging: receiving a message, transforming it, and publishing the result.

[source,java]
----
@Incoming("prices-in")
@Outgoing("prices-out")
public BigDecimal convertPrice(long priceInCents) {
    return BigDecimal.valueOf(priceInCents, 2);
}
----

=== 2. OrderResource - Programmatic Message Emission

Shows how to use `@Emitter` to programmatically send messages from a REST endpoint.

[source,java]
----
@Inject
@Channel("orders")
Emitter<Order> orderEmitter;

@POST
public Response createOrder(Order order) {
    orderEmitter.send(order);
    return Response.accepted().build();
}
----

=== 3. PaymentRequestHandler - Async Payment Processing with Acknowledgment

Demonstrates manual message acknowledgment while extending the store app's payment step.

[source,java]
----
@Incoming("orders")
@Outgoing("paid-orders")
@Acknowledgment(Acknowledgment.Strategy.MANUAL)
public CompletionStage<Order> requestPayment(Message<Order> message) {
    Order order = message.getPayload();
    return CompletableFuture.supplyAsync(() -> {
        order.setStatus(OrderStatus.PAID);
        return order;
    }).thenCompose(paidOrder -> message.ack().thenApply(v -> paidOrder));
}
----

This handler represents the order service handing work off asynchronously. The equivalent payment authorization logic is now shown in the dedicated `../payment` service example.

=== 4. OrderFilter - Stream Filtering

Shows how to filter message streams using reactive operators.

[source,java]
----
@Incoming("paid-orders")
@Outgoing("large-orders")
public PublisherBuilder<Order> filterLargeOrders(PublisherBuilder<Order> orders) {
    return orders.filter(order -> order.getTotalPrice().compareTo(BigDecimal.valueOf(1000.00)) > 0);
}
----

The example uses in-memory connectors for development. See `microprofile-config.properties` for Kafka configuration examples.

No explicit connector configuration is required for the default demo. The `order` example uses internal channels inside the same application, and the payment gateway URL is supplied through MicroProfile Config.

== Testing the Example

=== Create an Order via REST API

[source,bash]
----
curl -X POST http://localhost:9080/order/api/orders \
  -H "Content-Type: application/json" \
  -d '{
    "orderId": 1001,
    "userId": 42,
    "totalPrice": 1500.00,
    "status": "CREATED"
  }'
----

Expected response:
[source,json]
----
{
  "message": "Order accepted and published for payment processing"
}
----

The order will be:
1. Sent to the `orders` channel via the `Emitter`
2. Consumed by `PaymentRequestHandler`
3. Handed off asynchronously toward the dedicated Chapter 13 `payment` service flow
4. Acknowledged and forwarded to the `paid-orders` channel
5. Filtered by `OrderFilter` when the total is greater than `$1000`
6. Published to the `large-orders` channel

== Key Concepts Demonstrated

=== Declarative Messaging

* `@Incoming` - Consumes messages from a channel
* `@Outgoing` - Produces messages to a channel
* Method chaining - Same method can have both annotations

=== Programmatic Control

* `@Channel` - Inject a channel
* `@Emitter` - Programmatically send messages
* Useful for REST endpoints, scheduled tasks, etc.

=== Message Acknowledgment

* `MANUAL` - Developer controls when to acknowledge
* `message.ack()` - Acknowledge successful processing
* `message.nack(throwable)` - Negative acknowledgment on failure
* `PaymentRequestHandler` acknowledges only after the async payment hand-off completes successfully

=== Reactive Operators

* `filter()` - Select messages based on predicate
* `map()` - Transform messages
* `peek()` - Observe messages without modifying
* `flatMap()` - Transform each message into a stream

=== Backpressure

Automatically handled by the Reactive Streams specification. The framework ensures producers do not overwhelm consumers.
