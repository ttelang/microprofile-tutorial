== Overview

This chapter demonstrates the implementation of *MicroProfile Health 4.0* specification in a Jakarta EE 10 application. The Catalog Service implements comprehensive health monitoring with three types of health checks: startup, readiness, and liveness checks.

MicroProfile Health provides a common REST endpoint format to determine whether a microservice is healthy, ready to handle requests, and properly started. This is essential for:

* Container orchestration platforms (Kubernetes, Docker Swarm, OpenShift)
* Load balancers routing traffic based on service health
* Monitoring systems tracking application health status
* Automated recovery and restart decisions

== MicroProfile Health

MicroProfile Health provides a standardized way to report the health of microservices. The specification defines three types of health checks:

=== Startup Health Check

The startup health check verifies that the database schema is accessible during application startup:

[source,java]
----
@Startup
@ApplicationScoped
public class ProductServiceStartupCheck implements HealthCheck {

    private static final Logger LOGGER = Logger.getLogger(ProductServiceStartupCheck.class.getName());

    @PersistenceContext
    private EntityManager entityManager;

    @Override
    public HealthCheckResponse call() {
        try {
            // Verify database schema exists by checking table accessibility
            entityManager.createNamedQuery("Product.findAll", Product.class)
                        .setMaxResults(1)
                        .getResultList();
            
            LOGGER.info("Startup check: Database schema verified successfully");
            return HealthCheckResponse.named("ProductServiceStartupCheck")
                    .up()
                    .withData("message", "Database schema initialized")
                    .build();
        } catch (Exception e) {
            LOGGER.log(Level.SEVERE, "Startup check failed: Database schema not accessible", e);
            return HealthCheckResponse.named("ProductServiceStartupCheck")
                    .down()
                    .withData("error", e.getMessage())
                    .build();
        }
    }
}
----

*Key Features:*
* Verifies database schema accessibility via named query
* Uses proper logging framework (`java.util.logging.Logger`)
* Provides diagnostic data for both success and failure cases
* Runs during application startup phase
* Critical for database-dependent applications

==== Readiness Health Check

The readiness health check verifies database connectivity and ensures the service is ready to handle requests:

[source,java]
----
@Readiness
@ApplicationScoped
public class ProductServiceHealthCheck implements HealthCheck {

    private static final Logger LOGGER = Logger.getLogger(ProductServiceHealthCheck.class.getName());

    @PersistenceContext
    EntityManager entityManager;

    @Override
    public HealthCheckResponse call() {
        if (isDatabaseConnectionHealthy()) {
            return HealthCheckResponse.named("ProductServiceReadinessCheck")
                    .up()
                    .build();
        } else {
            return HealthCheckResponse.named("ProductServiceReadinessCheck")
                    .down()
                    .build();
        }
    }

    private boolean isDatabaseConnectionHealthy(){
        try {
            // Lightweight query that doesn't depend on specific data existing
            entityManager.createNamedQuery("Product.findAll", Product.class)
                        .setMaxResults(1)
                        .getResultList();
            return true;
        } catch (Exception e) {
            LOGGER.log(Level.WARNING, "Database readiness check failed", e);
            return false;
        }
    }
}
----

*Key Features:*
* Tests actual database connectivity via EntityManager
* Uses named query that doesn't depend on specific data existing
* Proper logging with `java.util.logging.Logger` at WARNING level
* Indicates service readiness to receive traffic
* Essential for load balancer health routing

==== Liveness Health Check

The liveness health check monitors system resources and memory availability:

[source,java]
----
@Liveness
@ApplicationScoped
public class ProductServiceLivenessCheck implements HealthCheck {

    private static final Logger LOGGER = Logger.getLogger(ProductServiceLivenessCheck.class.getName());

   @Override
   public HealthCheckResponse call() {
       Runtime runtime = Runtime.getRuntime();
       long maxMemory = runtime.maxMemory();
       long allocatedMemory = runtime.totalMemory();
       long freeMemory = runtime.freeMemory();
       long usedMemory = allocatedMemory - freeMemory;
       long availableMemory = maxMemory - usedMemory;

       long threshold = 100 * 1024 * 1024; // threshold: 100MB

       HealthCheckResponseBuilder responseBuilder = HealthCheckResponse.named("systemResourcesLiveness")
            .withData("FreeMemory", freeMemory)
            .withData("MaxMemory", maxMemory)
            .withData("AllocatedMemory", allocatedMemory)
            .withData("UsedMemory", usedMemory)
            .withData("AvailableMemory", availableMemory);

        if (availableMemory > threshold) {
            responseBuilder = responseBuilder.up();
        } else {
            responseBuilder = responseBuilder.down();
        }

        return responseBuilder.build();
    }
}
----

*Key Features:*
* Monitors JVM memory usage and availability
* Uses fixed 100MB threshold for available memory
* Provides comprehensive memory diagnostics via `.withData()` method
* Uses proper logging framework for consistency
* Indicates if application should be restarted

==== Health Check Endpoints

The health checks are accessible via standard MicroProfile Health endpoints:

* `/health` - Overall health status (all checks)
* `/health/live` - Liveness checks only
* `/health/ready` - Readiness checks only
* `/health/started` - Startup checks only

Example health check response:
[source,json]
----

  "status": "UP",
  "checks": [

==== Health Check Benefits

The comprehensive health monitoring provides:

* *Startup Validation*: Ensures all dependencies are initialized before serving traffic
* *Readiness Monitoring*: Validates service can handle requests (database connectivity)
* *Liveness Detection*: Identifies when application needs restart (memory issues)
* *Operational Visibility*: Detailed diagnostics for troubleshooting
* *Container Orchestration*: Kubernetes/Docker health probe integration
* *Load Balancer Integration*: Traffic routing based on health status

==== Health Check Implementation Best Practices

The health check implementations in this project follow MicroProfile Health best practices:

*1. Lightweight Checks*
* Readiness and startup checks execute quickly (< 1 second)
* Use `setMaxResults(1)` to limit query result size
* Avoid full table scans or expensive operations

*2. Proper Logging*
* All checks use `java.util.logging.Logger` instead of `System.err`
* Appropriate log levels: INFO for success, WARNING for readiness failures, SEVERE for startup failures
* Detailed error messages for troubleshooting

*3. No Hardcoded Dependencies*
* Startup and readiness checks use `createNamedQuery("Product.findAll").setMaxResults(1)`
* Don't depend on specific data existing (e.g., avoiding `find(Product.class, 1L)`)
* Gracefully handle empty databases

*4. Diagnostic Data*
* Startup check includes success/failure messages via `.withData()`
* Liveness check provides comprehensive memory metrics
* Enables troubleshooting without log access

*5. Correct Import Statements*
* Use `org.eclipse.microprofile.health.Startup` (not `jakarta.ejb.Startup`)
* Use `org.eclipse.microprofile.health.Liveness`
* Use `org.eclipse.microprofile.health.Readiness`

==== Kubernetes Integration

The health checks are designed for Kubernetes probe integration:

[source,yaml]
----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: catalog-service
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: catalog
        image: catalog-service:1.0
        ports:
        - containerPort: 5050
        
        # Startup probe - checked first
        startupProbe:
          httpGet:
            path: /health/started
            port: 5050
          initialDelaySeconds: 10
          periodSeconds: 5
          failureThreshold: 30  # 150 seconds max startup time
        
        # Liveness probe - determines restart
        livenessProbe:
          httpGet:
            path: /health/live
            port: 5050
          periodSeconds: 10
          timeoutSeconds: 3
          failureThreshold: 3
        
        # Readiness probe - determines traffic routing
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 5050
          periodSeconds: 5
          timeoutSeconds: 2
          failureThreshold: 2
----

== Server Configuration

To enable MicroProfile Health in Liberty, add the `mpHealth` feature to your `server.xml`:

[source,xml]
----
<server description="MicroProfile Tutorial Liberty Server">
    <featureManager>
        <platform>jakartaEE-10.0</platform>
        <platform>microProfile-6.1</platform>
        <!-- Base features -->
        <feature>restfulWS</feature>
        <feature>jsonp</feature>
        <feature>jsonb</feature>
        <feature>cdi</feature>
        <feature>persistence</feature>  <!-- Required for JPA health checks -->
        <feature>jdbc</feature>         <!-- Required for database health checks -->
        
        <!-- MicroProfile Health -->
        <feature>mpHealth</feature>     <!-- Enables health check endpoints -->
    </featureManager>

    <httpEndpoint httpPort="${default.http.port}" httpsPort="${default.https.port}"
                  id="defaultHttpEndpoint" host="*" />
    <webApplication location="catalog.war" contextRoot="${app.context.root}"/>
</server>
----

*MicroProfile Health Feature:*

* `mpHealth` - Implements health check endpoints for monitoring (startup, readiness, liveness checks)
* Provides standard endpoints at `/health`, `/health/live`, `/health/ready`, and `/health/started`
* No additional configuration required - health checks are automatically discovered via CDI

IMPORTANT: MicroProfile Health endpoints are registered at the **root server level** (e.g., `/health`), not under the application context root. Even though the application's REST API uses `/catalog` as the context root, health endpoints bypass this and are available directly at `/health/*`. This is standard MicroProfile Health behavior in Liberty.

NOTE: For complete server.xml configuration including datasource setup and Derby database configuration, refer to link:../../chapter03/catalog/README.adoc#liberty-server-configuration[Chapter 3 Liberty Server Configuration].

== Troubleshooting

=== Health Check Issues

*Health endpoint returns 503 (Service Unavailable):*

This indicates one or more health checks are failing. Check individual endpoints to identify which check is down:

[source,bash]
----
# Check individual health check types
curl http://localhost:5050/health/started  # Startup check
curl http://localhost:5050/health/ready    # Readiness check
curl http://localhost:5050/health/live     # Liveness check
----

*Startup check fails with "Database schema not accessible":*

* Verify the database is running and accessible
* Check the persistence.xml configuration
* Ensure the datasource is properly configured in server.xml
* Review link:../../chapter03/catalog/README.adoc#database-configuration[Chapter 3 Database Configuration]

*Readiness check fails with "Database readiness check failed":*

* Check database connectivity
* Verify the EntityManager is properly injected
* Ensure the named query "Product.findAll" exists in the Product entity
* Check server logs for detailed error messages

*Liveness check fails (low memory):*

* The liveness check fails when available memory drops below 100MB
* Review JVM memory settings: `-Xmx` and `-Xms` parameters
* Check for memory leaks in the application
* Consider increasing the memory threshold in the liveness check

*Health checks not appearing:*

* Ensure `mpHealth` feature is enabled in server.xml
* Verify health check classes are annotated with `@Startup`, `@Readiness`, or `@Liveness`
* Check that health check classes implement `HealthCheck` interface
* Ensure classes are discovered by CDI (in a CDI-enabled package)

*Wrong import causing compilation errors:*

Common mistake: Using `jakarta.ejb.Startup` instead of `org.eclipse.microprofile.health.Startup`

Correct imports:
[source,java]
----
import org.eclipse.microprofile.health.Startup;   // Correct
import org.eclipse.microprofile.health.Readiness; // Correct
import org.eclipse.microprofile.health.Liveness;  // Correct
import org.eclipse.microprofile.health.HealthCheck;
import org.eclipse.microprofile.health.HealthCheckResponse;
----
